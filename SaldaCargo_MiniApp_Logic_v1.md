# SaldaCargo — Подробная логика MiniApp v1.0

**Назначение:** Mini App в мессенджере MAX. Поставщик данных от ролей, которые работают «в полях»: водитель, механик, админ.
**Платформа:** Next.js SPA + MAX Mini App SDK, render внутри MAX.
**Принцип:** одна кодовая база, три UX-режима по `user.role`. Переключение мгновенное (если пользователь имеет несколько ролей).
**Дата:** 28.04.2026

---

## 0. Общие принципы для всех ролей

### 0.1. Авторизация и стартовый экран

При первом открытии приложения:

1. MAX SDK возвращает `max_user_id` и подпись.
2. Backend проверяет подпись, ищет `users.max_user_id`. Если не найден — экран «Доступ запрещён, обратитесь к администратору». Никаких самостоятельных регистраций.
3. Если найден — JWT cookie сохраняется, грузится домашний экран по роли.

Если у пользователя несколько ролей (например, Вова — `driver` и `loader`), показывается переключатель ролей на главном экране.

### 0.2. Офлайн-режим и идемпотентность

Это критически важно — водители ездят в межгород, механики работают в гараже с плохим сигналом.

**Правила:**

- Все мутации (создание заказа, расхода, запись таймера) сохраняются в IndexedDB сразу при сабмите формы. UI показывает «✓ Сохранено» немедленно.
- Параллельно отправляется запрос на сервер. При ошибке сети — попадает в очередь синхронизации.
- Каждая мутация несёт `idempotency_key` (UUID v4, генерится при открытии формы).
- При появлении сети очередь автоматически проталкивается. Дубли сервер отбрасывает по ключу.
- Бейдж в шапке: «📶 Онлайн / 🔄 3 в очереди / 📵 Офлайн».

### 0.3. Фото и Storage

- Все фото (одометры, чеки, дефекты) сжимаются на клиенте до ~1 МБ (canvas + WebP).
- Загрузка прямо в Supabase Storage с подписанным URL.
- При офлайне фото лежит в IndexedDB как Blob, заливается при появлении сети.
- В БД хранится только URL, не Base64.

### 0.4. Уведомления через MAX

Бот в MAX отправляет push-сообщения при ключевых событиях:

- Водителю: «Рейс утверждён», «Рейс возвращён на доработку», «ЗП за месяц начислена».
- Механику: «Новый наряд», «Запчасть пришла на склад».
- Админу: «Водитель завершил рейс», «Просрочка по клиенту X», «Касса > 500 000₽».

Уведомления — через MAX Bot API, не через Supabase Edge.

### 0.5. Общая навигация

Внизу экрана — таб-бар (3-4 пункта в зависимости от роли). Сверху — шапка с именем пользователя, ролью и индикатором сети. Переход между разделами без перезагрузок (SPA).

## 1. РОЛЬ: ВОДИТЕЛЬ

### 1.1. Назначение роли

Поставщик данных с рейса. Ничего не апрувит, ничего не считает финансово. Только фиксирует факты: что повёз, сколько взял денег, кто заплатил, какие были расходы. Все его записи попадают в БД со статусом `lifecycle: draft` и не влияют на P&L до апрува админа.

### 1.2. Структура навигации

Таб-бар (4 пункта):

- 🏠 **Главная** — текущий рейс или старт нового
- 📋 **Рейсы** — история своих рейсов
- 💰 **Финансы** — подотчёт + ЗП
- 👤 **Профиль** — настройки

### 1.3. Главная (без активного рейса)

**Что видно:**

- Приветствие: «Привет, Вова»
- Карточка «Мой подотчёт»: текущий остаток наличных у водителя (рассчитывается из транзакций). Например: «На руках: 10 100 ₽».
- Карточка «ЗП за месяц»: накопленная сумма по `trip_orders.driver_pay` за текущий месяц по рейсам со статусом `approved`. Если рейсы только в `draft` — показывается отдельно «В черновиках: 5 350 ₽».
- Большая кнопка `[🚚 Начать рейс]`.
- Список «Последние 3 рейса» с короткой сводкой: дата, машина, выручка, статус (черновик / на ревью / утверждён / возвращён).

**Кнопки и действия:**

- `[🚚 Начать рейс]` → переход на экран выбора машины и грузчика.
- Тап на карточку «Подотчёт» → детализация: список приходов (наличные с заказов) и расходов (ГСМ, мойка), итоговый остаток. Кнопка `[📤 Сдать кассу]` отсутствует — инкассацию инициирует админ, не водитель.
- Тап на карточку «ЗП» → переход на экран «Финансы».

### 1.4. Экран «Начать рейс»

**Поля формы:**

1. **Машина** — dropdown активных машин (`assets.status = active`). По умолчанию подставляется `users.current_asset_id` (закреплённая машина). Если водитель меняет машину — фиксируется в логе.
2. **Тип рейса** — radio: `local` (по городу), `intercity` (межгород), `moving` (переезд), `hourly` (почасовой). Влияет на подсказки ЗП и шаблоны.
3. **Грузчик** — dropdown пользователей с ролью `loader` или с дополнительной ролью грузчика. Опция «Без грузчика». Можно выбрать несколько (на переезды).
4. **Стартовый одометр** — число.
5. **Фото одометра** — обязательно для Газелей (asset_type.requires_odometer_photo = true), опционально для Валдаев (есть GPS).

**Кнопки:**

- `[▶ Поехали]` — создаёт `trip` со статусом `in_progress`, `lifecycle: draft`. Переход на экран активного рейса.
- `[Отмена]` — назад.

**Валидация:**

- Если есть незавершённый рейс этого водителя — кнопка «Поехали» заблокирована, показывается баннер «У вас есть активный рейс №247, завершите его сначала».
- Стартовый одометр должен быть ≥ последнего сохранённого `assets.odometer_current`. Если меньше — модалка «Одометр меньше предыдущего (340 100). Продолжить? (возможно, был сброс)».

### 1.5. Главная (с активным рейсом)

Когда есть `trip` со статусом `in_progress`, главная превращается в экран рейса.

**Что видно:**

- Шапка рейса: «Рейс №247 · Газель 866 · Вова + Серёга · идёт 3ч 47м»
- Кнопка `[📷 Сменить грузчика]` — на случай, если в середине дня грузчик ушёл / пришёл другой. Создаёт запись в логе экипажа, не меняет историю.
- Сводка по рейсу (live):
  - Заказов: 3
  - Выручка: 10 700 ₽
  - ЗП водителя: 3 650 ₽
  - ЗП грузчика: 3 650 ₽
  - Расходы: 0 ₽
- Список заказов (карточками, новые сверху). Каждая карточка: клиент, сумма, способ оплаты, ЗП водителя, иконка статуса оплаты (✅ оплачено / ⏳ долг).
- Список расходов (карточками).
- Две большие кнопки: `[➕ Заказ]` `[💸 Расход]`
- Кнопка внизу: `[🏁 Завершить рейс]`.

**Действия:**

- Тап на карточку заказа → детали + кнопка «Редактировать» (только пока рейс не завершён).
- Тап на карточку расхода → детали + редактирование.
- Долгий тап → удалить (с подтверждением). Удаление возможно только до завершения рейса.

### 1.6. Экран «Добавить заказ»

**Поля:**

1. **Клиент** — autocomplete по `counterparties.name` + опция «Без клиента (б/н)» + опция `[+ Новый клиент]` (открывает мини-форму: название, телефон, тип — клиент/поставщик).
2. **Описание** (опционально) — короткий текст: «Переезд», «Доставка плитки».
3. **Сумма** — число в рублях.
4. **ЗП водителя** — число с **подсказкой серым**: «~810 (30%)». Подсказка вычисляется по `payroll_rules` для текущего `trip_type` и `asset_type`. Водитель видит цифру, но вводит свою. Может быть 0.
5. **ЗП грузчика** — поле появляется, только если в рейсе есть `loader_id`. Аналогичная подсказка.
6. **Способ оплаты** — 5 кнопок-иконок:
   - 💵 **Наличные** (cash)
   - 📱 **QR на р/с** (qr) — деньги мгновенно на счёт ИП
   - 🏦 **Безнал по счёту** (bank_invoice) — клиент заплатит позже
   - ⏳ **Долг наличными** (debt_cash) — отдаст потом наличкой
   - 💳 **На карту водителя** (card_driver) — учётно как нал, физически на карту

**Кнопки:**

- `[✅ Добавить заказ]` — сохранение. Создаётся `trip_orders` + связанная `transaction` (income, draft).
- `[Отмена]` — назад.

**Логика подсказки ЗП:**

- При вводе суммы в поле «Сумма» — пересчитывается подсказка под полем ЗП.
- При выборе типа рейса (на старте рейса) подбирается базовое правило.
- Подсказка не блокирует ввод. Водитель волен ввести 1 000 при сумме 2 000 — это уйдёт админу на ревью с подсветкой.

**Settlement_status:**

- `cash`, `qr`, `card_driver` → `completed` (деньги уже у кого-то)
- `bank_invoice`, `debt_cash` → `pending` (создаётся дебиторка)

### 1.7. Экран «Добавить расход»

**Поля:**

1. **Категория** — dropdown с быстрыми кнопками: ГСМ / Мойка / Стоянка / Мелкий ремонт / Прочее.
2. **Сумма** — число.
3. **Способ оплаты** — 3 варианта:
   - 💵 Наличные (списывается с подотчёта водителя)
   - 💳 Карта водителя (учётно = наличные)
   - ⛽ Топливная карта Опти24 (только для категории FUEL — отдельный поток)
4. **Описание** (опционально).
5. **Фото чека** — опционально, но рекомендуется (для категорий REPAIR обязательно).

**Кнопки:**

- `[✅ Добавить расход]`
- `[Отмена]`

**Особенность ГСМ:**

- Если водитель выбирает «Топливная карта Опти24» — расход НЕ создаётся в этом потоке. Он автоматически прилетит из синхронизации с API Опти24 (auto-approved). Показывается надпись: «Заправки по картам Опти24 загружаются автоматически, ничего вводить не нужно».

### 1.8. Экран «Завершить рейс»

**Поля:**

1. **Конечный одометр** — число. Валидация: > стартового.
2. **Фото одометра** — обязательно для Газелей.
3. **Заметка для админа** (опционально) — свободный текст.

**Что видно — финальная сводка:**

- Заказов: 6 (с разбивкой по способам оплаты: 5 нал, 1 долг)
- Выручка: 16 050 ₽
- ЗП водителя: 5 350 ₽ (33%)
- ЗП грузчика: 4 950 ₽ (31%)
- Расходы: 1 000 ₽
- **Прибыль рейса:** 4 750 ₽
- Пробег: 60 км
- ⏳ Долги: 700 ₽ (1 клиент: «б/н»)

**Кнопки:**

- `[📤 Отправить на ревью]` — `trip.status = completed`, `lifecycle: draft`, обновляется `assets.odometer_current` (триггер ТО), для Валдаев — фоновый запрос в Wialon для верификации пробега.
- `[← Вернуться к рейсу]` — назад.

**После отправки:** редирект на главную, баннер «Рейс №247 ушёл на ревью. Ожидайте подтверждения».

### 1.9. Раздел «Рейсы»

История своих рейсов с фильтром по статусу:

- Все
- 📝 Черновики (in_progress)
- ⏳ На ревью (completed + draft)
- ✅ Утверждённые (approved)
- ↩️ Возвращённые на доработку
- ❌ Отменённые (cancelled)

**Карточка рейса:** дата, машина, заказов, выручка, прибыль, статус.
**Тап:** детальный экран рейса со всеми заказами, расходами, фото одометров.

**Возвращённый рейс:**
Если админ нажал «Вернуть на доработку», рейс возвращается со статусом `lifecycle: returned`, водитель видит красный баннер сверху с комментарием админа («Проверь сумму по заказу №3»). После исправления — снова `[📤 Отправить на ревью]`.

### 1.10. Раздел «Финансы»

Две вкладки:

**Вкладка 1: Подотчёт**

- Текущий остаток (вычисляется по транзакциям подотчётного кошелька)
- История движений: приходы (нал от клиентов) и расходы (ГСМ, инкассации)
- Каждая запись с датой, суммой, описанием, источником (рейс №X)

**Вкладка 2: ЗП**

- Текущий месяц: накоплено к выплате, выдано авансами, к доплате
- История по рейсам: дата, машина, заработано
- Прошлые месяцы: closed periods с финальными суммами и датой выплаты

### 1.11. Раздел «Профиль»

- ФИО, дата рождения, телефон
- Закреплённая машина
- Кнопка `[Связаться с админом]` (открывает чат в MAX)
- Кнопка `[Выйти]`

## 2. РОЛЬ: МЕХАНИК

### 2.1. Назначение роли

Поставщик данных из СТО. Фиксирует факт ремонта: что чинил, сколько по факту времени, что списал со склада. Финансово изолирован — не видит ни цен для клиента, ни своей ЗП за конкретный наряд. ЗП считается по факту в WebApp админом и приходит в раздел «Финансы».

### 2.2. Структура навигации

Таб-бар (4 пункта):

- 🏠 **Главная** — активные наряды + очередь
- 🔧 **Наряды** — история своих закрытых нарядов
- 📦 **Склад** — остатки запчастей (read), заявки на закупку
- 👤 **Профиль**

### 2.3. Главная

**Что видно:**

- Шапка: «Привет, Вадик»
- Карточка с активным таймером (если есть): «Идёт работа: Замена сцепления, Газель 099 · 1ч 23м». Кнопки `[⏸ Пауза]` и `[⏹ Стоп]`.
- Секция **«В работе»** — список нарядов, которые механик взял (`assigned_mechanic_id = me`, `status = in_progress`).
- Секция **«Очередь»** — список нарядов без исполнителя (`status = created, assigned_mechanic_id IS NULL`). Каждый — кнопка `[Взять в работу]`.
- Кнопка `[+ Новый наряд]` — внизу, для случая, когда механик сам начинает работу (например, машина пришла без согласования с админом).

**Карточка наряда:** машина (своя/клиентская), краткое описание проблемы, дата создания, приоритет (если задан).

### 2.4. Экран «Взять в работу» / «Новый наряд»

Если механик жмёт `[Взять в работу]` на существующем наряде — он сразу попадает в экран наряда (см. 2.6). Если жмёт `[+ Новый наряд]` — заполняет форму:

**Поля:**

1. **Тип машины** — radio: «Наш автопарк» / «Клиентская».
2. **Машина** — если «Наш» — dropdown активных. Если «Клиентская» — поля «Марка», «Модель», «Госномер», «ФИО клиента», «Телефон». Создаётся `counterparty` с типом `client`.
3. **Текущий одометр** (для своих машин — обновит `assets.odometer_current`).
4. **Описание проблемы** — свободный текст.
5. **Фото проблемы** (опционально).

**Кнопки:**

- `[✅ Создать и взять]` — создаёт `service_order` со статусом `in_progress`, сразу присваивается этому механику.
- `[Отмена]`

### 2.5. Справочник работ (work_catalog)

Это новая таблица, которой не было в схеме v2.3. Её нужно добавить:

```
work_catalog:
  id, code, name (e.g. "Замена сцепления Газель"),
  applicable_asset_type_ids (массив),
  norm_minutes (норма времени),
  default_price_client (для клиентских машин),
  internal_cost_rate (ставка для своих, может быть 0),
  is_active
```

Заполняется админом в WebApp. На MVP можно завести 20-30 ключевых работ, дальше пополнять по ходу.

### 2.6. Экран наряда (главный экран механика)

**Шапка:**

- Номер наряда: «Наряд №142»
- Машина: «Газель 099 · одометр 340 580»
- Время с момента взятия в работу.

**Секция «Работы»:**
Список добавленных работ. Каждая — карточка:

- Название работы (из каталога или свободная)
- Норма времени: 240 мин
- Факт: 1ч 47м (live-таймер если запущен)
- Состояние: ⏸ Не начата / ▶ Идёт / ⏸ На паузе / ✅ Завершена
- Кнопки управления:
  - Если не начата: `[▶ Старт]`
  - Если идёт: `[⏸ Пауза]` `[⏹ Завершить]`
  - Если на паузе: `[▶ Продолжить]` `[⏹ Завершить]`
  - Если завершена: `[✏️ Изменить факт]` (с подтверждением, для исправлений)

**Только один таймер может быть запущен одновременно.** При попытке стартовать вторую работу — модалка «Сначала поставьте на паузу текущую: «Замена сцепления»».

**Секция «Запчасти»:**
Список списанных деталей. Карточка: название, количество, иконка (✓ списано / ⏳ резерв / ❌ нет на складе).

**Секция «Дефекты»:**
Найденные в процессе дефекты — превью фото + комментарий.

**Внизу экрана:**

- `[+ Добавить работу]`
- `[+ Запчасть со склада]`
- `[📸 Найден дефект]`
- `[🏁 Завершить наряд]`

### 2.7. Экран «Добавить работу»

**Поля:**

1. **Работа** — autocomplete по `work_catalog`, отфильтровано по `applicable_asset_type_ids` для текущей машины. Опция `[+ Свободная работа]` — поля «Название», «Описание». Помечается `requires_admin_review = true`, чтобы админ потом добавил её в каталог.
2. **Заметка** (опционально).

**Кнопки:**

- `[✅ Добавить]` — создаёт `service_order_works` со статусом `pending`. Цена для клиента и норма времени фиксируются СНИМКОМ из каталога на момент добавления — потом админ может поменять каталог, но в этом наряде останется как было.
- `[Отмена]`

### 2.8. Логика таймера

**Старт:**

- Создаётся `work_time_logs` запись: `started_at = now()`, `status = running`.
- В шапке приложения появляется плашка «Идёт работа: ... · MM:SS».

**Пауза:**

- Текущая запись `work_time_logs` закрывается: `stopped_at = now()`, `status = paused`.
- Открывается модалка «Причина паузы»:
  - Обед / Перерыв
  - Жду запчасть
  - Другая работа
  - Конец смены
  - Другое (текстом)
- Причина пишется в `work_time_logs.pause_reason`.

**Продолжить:**

- Создаётся НОВАЯ запись `work_time_logs` (так проще, чем апдейтить старую). Не одна длинная запись с паузами, а несколько коротких сегментов.

**Завершить:**

- Закрывается текущая запись.
- `service_order_works.status = completed`, `actual_minutes = SUM(stopped - started)` по всем сегментам.
- Если `actual_minutes > norm_minutes × 1.5` — пометка для ревью админа.

**Особый случай — работа закрыта без таймера:**
Иногда механик забывает нажать «Старт». На экране работы есть кнопка `[Я уже сделал, ввести время вручную]` — открывает поле «Сколько потратил по факту». Пишется в `actual_minutes`, в `work_time_logs` создаётся одна запись с пометкой `manual_entry = true`. Админ видит эту пометку при ревью.

### 2.9. Экран «Запчасть со склада»

**Поля:**

1. **К работе** — dropdown работ из этого наряда (если их больше одной).
2. **Запчасть** — autocomplete по `parts`, показывает остаток на складе и цену закупки. Если остаток 0 — позиция помечена «Нет на складе».
3. **Количество** — число.

**Кнопки:**

- `[Отмена]`

## 3. РОЛЬ: АДМИН (мобильный пульт)

### 3.1. Назначение роли

Под этой ролью работаешь ты и Владелец, **когда вы не за компьютером**. Админ в MiniApp — это «карманный пульт» для трёх ситуаций:

1. Инкассация наличных у водителя на базе.
2. Погашение дебиторки (клиент перевёл).
3. Фиксация спонтанных расходов (купил канцелярию).

**MiniApp админа НЕ дублирует Dashboard.** Нет ревью смен (это в Web), нет аналитики, нет редактирования справочников. Только быстрые финансовые операции.

### 3.2. Структура навигации

Таб-бар (3 пункта):

- 🏠 **Главная** — быстрые действия + дневная сводка
- 💰 **Деньги** — балансы кошельков + история операций
- 👤 **Профиль**

### 3.3. Главная

**Что видно:**

- Сводка «Сегодня»: выручка, ФОТ, прибыль, машин на линии — компактные плашки.
- Алерты (если есть):
  - «🔴 3 рейса ждут ревью» (тап → переход на WebApp через openLink)
  - «🟠 Просрочка по «Левше» 5 дней — 12 000 ₽»
  - «🟡 Касса > 500 000 ₽, пора в банк»
- 4 большие кнопки:
  - `[💵 Забрать выручку]` — инкассация
  - `[+ Приход]` — погашение долга или прочий приход
  - `[− Расход]` — спонтанный расход
  - `[🔄 Перевод между кошельками]` — например, снял с р/с положил в кассу

### 3.4. Экран «Забрать выручку» (инкассация)

**Шаг 1: выбор водителя**

- Список всех водителей с ненулевым подотчётом, отсортирован по убыванию суммы.
- Каждая строка: имя, текущий остаток в подотчёте, дата последнего рейса.
- Тап на строку → переход к шагу 2.

**Шаг 2: подтверждение суммы**

- Шапка: «У Вовы на руках: 15 400 ₽»
- Поле ввода: сколько забираем сейчас. По умолчанию подставляется полная сумма. Можно изменить (например, оставить водителю 400 ₽ на завтра).
- Поле «Куда» — radio:
  - Касса офиса (по умолчанию)
  - Сразу на р/с (если планируешь сейчас в банк)
  - Личный карман Владельца (если Владелец забрал на свои нужды — учётно это «дивиденды ИП»)
- Заметка (опционально).

**Кнопки:**

- `[✅ Забрать 15 000 ₽]`
- `[Отмена]`

**Что в БД:**

- Создаётся `transaction` типа `transfer`: `from_wallet = driver_vova_wallet`, `to_wallet = cash_office`, amount = 15 000, `lifecycle: approved` (админ создал, сразу утверждено), `transaction_type = cash_collect`.
- Подотчёт водителя пересчитывается, касса увеличивается.
- В MAX-чате водителю приходит уведомление: «Админ забрал 15 000 ₽ из вашего подотчёта. Остаток: 400 ₽».

### 3.5. Экран «Приход»

Универсальная форма для любого прихода денег извне.

**Тип прихода — radio:**

1. **Погашение долга клиента** (самый частый случай)
2. **Прочий приход** (нашли деньги в столе, возврат от поставщика и т.п.)

**Если «Погашение долга»:**

- Список открытых дебиторок (`transactions WHERE settlement = pending AND direction = income`), сортировка по дате просрочки. Каждая: клиент, сумма, дни просрочки.
- Поиск/фильтр.
- Тап на запись → шаг 2:
  - Сколько пришло (по умолчанию — полная сумма; можно частично)
  - Куда (р/с / касса / на карту → потом инкассируется)
  - Дата прихода (по умолчанию сегодня)
- `[✅ Принять]` → находит исходную транзакцию-долг, переводит её `settlement = completed` (если оплачено полностью) или создаёт частичный платёж + остаток долга. Деньги начисляются на выбранный кошелёк.

## 4. КРОСС-РОЛЕВЫЕ ВЗАИМОДЕЙСТВИЯ

### 4.1. Изменение справочников Web → Mini

Когда в WebApp админ:

- Добавил машину → она появляется в dropdown водителей и механиков.
- Добавил работу в `work_catalog` → доступна механикам.
- Изменил `payroll_rules` (подсказку ЗП) → новая цифра показывается в подсказках при создании заказа.
- Добавил/изменил клиента → доступен в autocomplete у водителей.

**Реализация:** Supabase Realtime — приложение подписано на ключевые таблицы и при изменении делает invalidate соответствующего кэша TanStack Query.

### 4.2. Конфликт «правка vs апрув»

Сценарий: водитель открыл свой рейс на правку (рейс был возвращён), а в этот момент админ заходит в WebApp и пытается апрувнуть. Что делать?

- В БД хранится `lifecycle_status` + `updated_at`.
- При сабмите правки клиент шлёт `If-Unmodified-Since: <updated_at>`.
- Сервер сверяет — если на сервере `updated_at` свежее → 409 Conflict.
- На клиенте показывается «Этот рейс был изменён администратором. Обновить?» с кнопкой обновления.

При апруве — то же самое: если водитель что-то изменил в момент апрува, админ получает «Рейс был изменён водителем. Просмотрите изменения».

### 4.3. Версионирование справочников

Цифры из справочников (нормы времени, цены работ, ставки ЗП) **фиксируются снимком при создании сущности**, не вычисляются на лету.

**Пример:**

- 1 марта в `work_catalog` цена «Замены сцепления» = 8000 ₽.
- 15 марта механик добавил эту работу в наряд №142 → в `service_order_works.price_client = 8000` (записалось как факт).
- 20 марта админ поменял цену в каталоге на 9500 ₽.
- В наряде №142 остаётся 8000 ₽ — как было на момент добавления.

То же для подсказок ЗП в заказах: подсказка считается на лету, но **введённая цифра** уходит в БД как факт.

### 4.4. Realtime-обновления Mini → Web

Это не строго логика MiniApp, но критично для системы:

- Водитель завершил рейс → у админа в WebApp в «Ревью смены» сразу появляется карточка (Supabase Realtime подписка на `trips` + lifecycle = draft + completed).
- Механик закрыл наряд → у админа появляется в «Ревью нарядов».
- Админ инкассировал → у водителя в MiniApp обновляется подотчёт без перезагрузки.

## 5. UX-принципы и отказы

### 5.1. Большие пальцы, узкий экран

- Минимальный размер кнопок — 48×48 px.
- Главные действия — большие кнопки, минимум 64 px высоты.
- Поля ввода крупные, цифровая клавиатура (`inputmode=decimal`) для сумм.
- Все важные действия требуют подтверждения (модалка), кроме реверсивных (например, добавить заказ — отменимо до завершения рейса).

### 5.2. Никаких удалений

В соответствии с принципом v2.3 — никаких DELETE.

- Заказ создан и сохранён → удалить можно только до отправки рейса на ревью. Удаление = soft-delete с `cancelled_reason`.
- Рейс отправлен на ревью → менять может только админ (через возврат на доработку).
- Транзакции от админа (приход, расход) — отменяются через создание `transaction-обратки` с пометкой «correction».

### 5.3. Что показываем при ошибках сети

- Бейдж в шапке: «📵 Офлайн».
- Все мутации идут в очередь, UI работает.
- При попытке открыть данные, которых нет в кэше — экран «Нет интернета. Покажу когда подключитесь» + кнопка `[Повторить]`.
- Нельзя начать новый рейс офлайн, если справочники не закэшированы (нужны машины, грузчики, клиенты). Проверка при первом онлайн-сеансе после установки приложения — фоновая загрузка всех справочников в IndexedDB.

### 5.4. Время и часовые пояса

- Все timestamps в БД в UTC (`TIMESTAMPTZ`).
- На клиенте — Asia/Yekaterinburg (Верхняя Салда, UTC+5).
- Даты в UI: «Сегодня», «Вчера», «19 апр», «19 апр 2025».
- Время: «18:30», «3ч 47м» для длительностей.
