# SaldaCargo — Подробная логика WebApp v1.0

**Назначение:** Web Dashboard. Полное оцифровывание компании — управление, контроль, аналитика, апрув данных от ролей в MiniApp.
**Платформа:** Next.js 15 (App Router) + React 19 + Tailwind + shadcn/ui + Tremor (графики).
**Принцип:** в WebApp работают `owner`, `admin`, `accountant`, `mechanic_lead`. Под одной учёткой может быть несколько ролей. UI адаптируется по роли — accountant видит read-only, mechanic_lead видит склад на запись и т.д.
**Дата:** 28.04.2026

---

## 0. Общие принципы

### 0.1. Авторизация и доступ

При первом заходе:

1. Логин через MAX OAuth (тот же бот, что и для MiniApp) → JWT.
2. Альтернатива на MVP — magic-link на email через Supabase Auth (для тех, у кого нет MAX, например бухгалтер на аутсорсе).
3. Backend проверяет роль. Если `driver/loader/mechanic` — редирект на специальную страницу «Этот раздел только для админов, используйте мобильное приложение».

**Роли в WebApp:**

- `owner` — Владелец. Видит всё.
- `admin` — ты. Видит всё, кроме настроек уровня собственности (продажа машины, создание юрлица — требует подтверждения owner).
- `accountant` — read-only + экспорт.
- `mechanic_lead` — старший механик. Доступ только к разделу «Гараж» с правом записи на склад и ведения нарядов. Финансы и автопарк — read-only без сумм.

**Top bar:**

- Логотип/название

- Меню пользователя: профиль, переключатель юрлица (если их несколько), выход

**Sidebar:** разделы. Состояние (свёрнуто/развёрнуто) запоминается в localStorage.

### 0.3. Глобальные элементы

- **Фильтр периода** — на каждом разделе с динамическими данными. Дефолт: текущий месяц. Опции: Сегодня / Эта неделя / Этот месяц / Прошлый месяц / Этот год / Произвольный.
- **Индикатор Realtime** — точка в углу таблиц («подключено / переподключение»). Ключевые таблицы (рейсы на ревью, наряды) обновляются без перезагрузки.
- **Хлебные крошки** на вложенных страницах: «Автопарк / Газель 866 / Аналитика».
- **Подтверждения опасных действий** — модалка с явным «Я понимаю последствия» (продажа машины, отмена транзакции, удаление сотрудника).

### 0.4. Подход к таблицам

Везде, где список длиннее 10 записей:

- Виртуализация строк (TanStack Table).
- Сортировка по колонкам (клик на заголовок).
- Фильтры в шапке колонок (выпадающий чек-лист для категорий, диапазон для чисел).
- Поиск по таблице.
- Экспорт в CSV/Excel.
- Сохранённые виды (опционально v2): «Мой фильтр Газели за последний месяц».

### 0.5. Realtime через Supabase

Подписки на ключевые таблицы:

- `trips` со статусом `draft+completed` → обновление списка ревью.
- `service_orders` со статусом `draft+completed` → обновление ревью нарядов.
- `transactions` свежие → обновление главной и финансов.
- `defect_log` → обновление бэклога.

Не подписываемся на справочники — там данные меняются редко, обновляются при ручной перезагрузке.

## 2. РЕВЬЮ СМЕНЫ

### 2.1. Назначение

**Точка контроля.** Здесь данные от водителей и механиков превращаются из `draft` в `approved`. До этого момента ничего не считается в P&L и в балансах. Это сердце ежедневной работы owner/admin.

### 2.2. Структура раздела

Две вкладки:

1. **Рейсы водителей** (по умолчанию)
2. **Наряды механиков**

И фильтр по дате: по умолчанию сегодня, можно листать по дням или выбрать диапазон. **Важно:** раздел не теряет историю — в любой день прошлого можно зайти и посмотреть, что и как было утверждено.

### 2.3. Вкладка «Рейсы водителей»

**Шапка дня:**

- Дата: 19 апреля 2026
- Статус: «12 записей от 2 машин · 25 450 ₽ выручки · Ожидает подтверждения / Утверждено в 23:14»
- Кнопка `[✅ Утвердить всё за день]` (только если есть драфты)

**Группировка по машинам.** Каждая машина — собственная карточка с таблицей.

**Карточка машины (например, Газель 866):**

```
🚚 Машина 866 · Вова + Серёга · 08:30–18:00 · 60 км
┌─────────────────────────────────────────────────────────────┐
│ # │ Клиент    │ Сумма │ ЗП Вод │  %  │ ЗП Гр │ Оплата    │
│ 1 │ Интерьер  │ 2 700 │ 1 000  │ 37% │ 1 000 │ 💵 Нал    │
│ 2 │ Переезд   │ 6 000 │ 2 000  │ 33% │ 2 000 │ 💵 Нал    │
│ ...                                                          │
│ 6 │ б/н    ⏳ │   700 │   200  │ 29% │     0 │ ⏳ Долг   │
├─────────────────────────────────────────────────────────────┤
│ Итого:        16 050 │ 5 350  │ 33% │ 4 950 │              │
│ ГСМ: 1 000  Прибыль: 4 750  Пробег: 60 км                  │
└─────────────────────────────────────────────────────────────┘
[📷 Фото одометров] [✅ Утвердить] [↩️ Вернуть] [✏️ Редактировать]
```

**Подсветка аномалий:**

- ⚠️ Жёлтый: % водителя выходит за коридор 25–40%.
- 🔴 Красный: GPS-пробег расходится с одометром > 10% (для Валдаев).
- ⚠️ Жёлтый: длительная пауза между заказами > 2ч без объяснения.
- ⚠️ Жёлтый: одометр конечный − начальный сильно расходится с расчётным (по сумме чеков ГСМ).

**Действия:**

`[📷 Фото одометров]` — модалка с двумя фото (старт/финиш) + GPS-данные для Валдаев. Можно сравнить визуально.

`[✅ Утвердить]` — открывает финальное подтверждение со сводкой. После подтверждения:

- `trip.lifecycle_status = approved`, `approved_by = me`, `approved_at = now()`.
- Все связанные `trip_orders` и `transactions` → `approved`.
- Транзакции с `settlement = pending` остаются `pending` (это дебиторка, которую закроем когда придут деньги).
- Подсчитанные суммы попадают в P&L и в балансы.

`[↩️ Вернуть]` — модалка «Что не так?». Поле комментария обязательно. Статус → `lifecycle: returned`. Водитель в MiniApp видит баннер с твоим комментарием.

`[✏️ Редактировать]` — раскрывает inline-редактирование таблицы. Можешь поменять цифры (например, исправить ЗП с 1300 на 1000, если водитель ошибся). Каждая правка пишется в `audit_log` с указанием, что и кто менял. После правки — кнопка `[💾 Сохранить и утвердить]`.

**Свернуть/развернуть:** карточки машин по дефолту развёрнуты. Если за день много машин — можно свернуть в сводку.

### 2.4. Вкладка «Наряды механиков»

Аналогично рейсам, но карточки — по нарядам.

**Карточка наряда:**

```
🔧 Наряд №142 · Газель 099 · Вадик · 09:00–13:12 · 4ч 12м
┌─────────────────────────────────────────────────────────────┐
│ Работа              │ Норма │ Факт │ Эффект.│ Стоимость   │
│ Замена сцепления    │ 240   │ 220  │ 109%   │ 8 000       │
│ Замена масла        │  30   │  35  │  86%   │ 1 000       │
│ Регулировка сцепл.  │  60   │  72  │  83%   │ 1 500       │
├─────────────────────────────────────────────────────────────┤
│ Запчасти:                                                   │
│ Сцепление в сборе LUK   1 шт   12 500 ₽                    │
│ Масло Castrol 5W40      4 л     2 800 ₽                    │
│ Прокладка картера       1 шт      350 ₽                    │
├─────────────────────────────────────────────────────────────┤
│ Дефекты найдены: 1 → [Сальник коленвала, фото]              │
│ Заметка от механика: «Колодки на след. ТО рекомендую»      │
└─────────────────────────────────────────────────────────────┘
Эта машина: [Своя]    Стоимость работ: 10 500 ₽    Запчасти: 15 650 ₽
[💰 Сформировать счёт] [✏️ Изменить ставку] [✅ Утвердить] [↩️ Вернуть]
```

**Действия:**

`[💰 Сформировать счёт]` — для **клиентских** машин. Открывает форму:

- Применить наценку: % или фиксированная сумма
- Скидка клиенту: %
- Итоговая сумма
- Способ оплаты: Нал в кассу / На р/с / В долг
- `[✅ Создать счёт]` → создаёт `transaction income, settlement: completed/pending`, привязанный к наряду.

`[✏️ Изменить ставку]` — для **своих** машин. Можешь задать ЗП механику за этот наряд. По умолчанию подставляется по правилу (норма-час × ставка), можно изменить вручную (как с водителями — ручной ввод первичен, формула — подсказка).

`[✅ Утвердить]` — наряд → `lifecycle: approved`. Запчасти со статусом `reserved` переходят в `consumed` (списываются окончательно). ЗП механика записывается в `payroll_periods`. Если машина клиентская — счёт уже создан и отмечен как доход.

`[↩️ Вернуть]` — на доработку с комментарием.

**Подсветка аномалий:**

- 🔴 Эффективность < 70% или > 150% (норма явно неправильная или работали без таймера).
- ⚠️ Запчасти на сумму > 30% от стоимости работ — обычно сигнал, что что-то не учтено.
- ⚠️ Свободная работа без `work_catalog_id` — пометка «добавить в каталог».

### 2.5. Архив утверждённого

В историческом режиме (выбрана прошедшая дата) — те же карточки, но без кнопок утверждения. Только `[👁 Подробнее]` и `[📋 Скопировать для аудита]`.

Если кто-то вспомнит «а что мы 5 марта утвердили по 866-й?» — заходишь в дату, видишь те же таблицы как они были в момент апрува.

### 2.6. Сводка дня в шапке

Над всеми карточками — итоговая полоска по выбранному дню:

- Выручка / ФОТ водителей / ФОТ грузчиков / ФОТ механиков / ГСМ / Запчасти / Прибыль
- Машин на линии / Нарядов закрыто / Долгов создано / Долгов погашено

Это та же сводка, что на Главной в зоне «Сегодня», но за конкретный выбранный день.

**Действия:**

- `[✏️ Редактировать]`
- `[🚫 Списать]` (с причиной — авария, износ; меняет статус на `written_off`, амортизация прекращается)
- `[💰 Продать]` — открывает форму продажи (см. 3.7)

#### Закладка 2: Аналитика 360

**Это ключевой экран всей системы.** Здесь видно, кормит ли машина компанию.

**Шапка с фильтром периода** (по умолчанию текущий месяц).

**Большие KPI (4 плашки):**

1. **Выручка** — сумма по `trip_orders.amount` всех approved рейсов на этой машине за период.
2. **Себестоимость** — все расходы, привязанные к машине: ГСМ + ЗП водителя + ЗП грузчиков + ремонты + амортизация + страховка (пропорционально периоду).
3. **Прибыль** — разница.
4. **Маржа** — прибыль / выручка %.

**Юнит-экономика 1 км:**

- Выручка / км
- Себестоимость / км (детализация ниже)
- Прибыль / км

**Декомпозиция себестоимости** (пирог Tremor):

- ГСМ X% (Y₽/км)
- ЗП водителя X%
- ЗП грузчика X%
- Ремонты X%
- Амортизация X%
- Страховка X%
- Прочее X%

**График «Прибыль по месяцам»** (бары за последние 12 мес) — видно сезонность и тренд.

**График «Загрузка по дням»** (heatmap календарь — сколько часов в работе каждый день за последние 30 дней).

**Сравнение с парком:**

- Среднее по типу (например, средняя прибыль/км по всем Валдаям): X
- Эта машина: Y (выше/ниже среднего)
- Кнопка `[Сравнить с другой машиной]` — модалка с выбором → side-by-side таблица KPI.

#### Закладка 3: Техсостояние

**Текущий пробег** + дата последнего апдейта.

**План ТО:**

- Регламенты из `maintenance_regulations` для этого `asset_type`:
  - Замена масла — каждые 10 000 км
  - Замена фильтров — каждые 20 000 км
  - Замена ремня ГРМ — каждые 60 000 км
  - и т.д.
- Для каждого: последняя замена (km, дата), следующая (km, дата прогноз), остаток.
- Полоса состояния: зелёная (>20% запаса), жёлтая (5-20%), красная (просрочено).

**Активные заявки на ремонт** — справа плитка.

- Список открытых нарядов (`service_orders` со статусом `created` или `in_progress`).
- Кнопка `[+ Создать заявку]` — форма создания нового наряда (открывает в разделе «Гараж» с предзаполненной машиной).

**Бэклог дефектов:**

- Список из `defect_log` со статусом `open` для этой машины.
- Каждый: фото, описание, срочность, кто нашёл, когда.
- Действия: `[Запланировать]` (статус → `planned`, можно привязать к будущему наряду), `[Отклонить]` (статус → `dismissed` с причиной).

**История ТО:** список всех закрытых нарядов с датой, видами работ, стоимостью.

#### Закладка 4: История рейсов

Таблица всех `trips` этой машины с фильтром по периоду.

Колонки: дата, водитель, грузчик, заказов, выручка, ЗП водителя, ГСМ, прибыль рейса, пробег, статус.

Тап на строку → детальный просмотр рейса (как в Ревью смены, но read-only).

Кнопка `[Экспорт CSV]`.

### 3.4. Создание машины

Форма (открывается из `[+ Добавить автомобиль]`):

**Шаг 1: Тип и номер**

- Тип (asset_type): dropdown
- Госномер
- VIN (опц.)
- Год выпуска
- Марка/модель

**Шаг 2: Эксплуатация**

- Текущий пробег
- Остаточная стоимость (₽)
- Оставшийся срок службы (мес) — амортизация автоматически = ост.стоимость / срок
- Статус (по умолчанию active)
- Юрлицо
- BU (направление)
- Закреплённый водитель (опц.)

**Шаг 3: Документы (опц.)**

- СТС, ПТС
- ОСАГО (номер, действует до)
- КАСКО

**Шаг 4: GPS (для Валдаев)**

- Wialon ID объекта

**Шаг 5: Проверка → `[✅ Создать]`**

После создания:

- `assets` запись со статусом active
- `wallets` не создаются (это не сотрудник)
- В `transactions` — initial entry с `transaction_type = initial_balance` для отражения остаточной стоимости в активах компании.

### 3.5. Продажа машины

Кнопка `[💰 Продать]` в детальном экране машины.

**Форма:**

- Дата продажи
- Покупатель (counterparty или новый)
- Сумма продажи
- Способ получения (р/с / нал / в долг)
- Причина продажи (свободный текст)
- Прикрепить договор / акт (опц., файл в Storage)

**Что происходит при подтверждении:**

- `assets.status = sold`, `assets.sold_at = now()`, `current_book_value = 0`.
- Создаётся `transaction income`, `category = ASSET_SALE`, сумма = сумма продажи.
- Создаётся `transaction` корректировки балансовой стоимости (она была 1 432 000, продали за 800 000 → разница 632 000 идёт в `category = ASSET_WRITE_OFF`).
- Машина пропадает из активной сетки автопарка, но остаётся доступна по фильтру «Проданные» и в исторических рейсах.

- Пробег
- Часы в работе
- Дни в ремонте
- КТГ (коэффициент технической готовности)
- Возраст
- Балансовая стоимость
- Доход на единицу балансовой стоимости (ROI)

Графики: bars по KPI рядом, линии прибыли по месяцам наложены.

Это инструмент для решений «продать ли Валдай 446».

## 4. ГАРАЖ / СТО

### 4.1. Назначение

Управление эффективностью механиков, складом запчастей, ценообразованием. Отдельное направление бизнеса, которое сейчас обслуживает свой парк, а в будущем — клиентов.

### 4.2. Структура раздела

6 подразделов (вкладки):

1. **Активные наряды** (kanban)
2. **Архив нарядов**
3. **Справочник работ**
4. **Склад запчастей**
5. **Эффективность механиков**
6. **Бэклог дефектов**

### 4.3. Подраздел 1: Активные наряды (Kanban)

Канбан-доска со статусами:

```
[Создан] → [В работе] → [Завершён] → [Утверждён] → [Счёт выставлен]
```

**Карточка наряда** на доске:

- Номер
- Машина (своя или клиентская — иконка)
- Краткое описание проблемы
- Механик (если назначен)
- Время с создания
- Цвет: своя / клиентская

**Drag-and-drop** между колонками — для админа. Например, перетащил в «Завершён» → как будто нажал «Утвердить».

**Фильтры:**

- Свои / Клиентские
- По механику
- По типу машины

**Кнопка `[+ Создать наряд]`** — форма (см. 4.4).

### 4.4. Создание наряда из WebApp

Используется когда:

- Планово (по регламенту ТО) — создаёшь заранее.
- По заявке от водителя (он в MiniApp написал заметку «стучит подвеска»).
- Клиентская машина пришла на ремонт.

**Поля:**

1. Тип машины: Наша / Клиентская
2. Машина: dropdown (для своих) или поля для клиентской (марка, номер, контактное лицо)
3. Описание проблемы
4. Приоритет: Низкий / Обычный / Срочный
5. Назначить механика (опц., можно оставить в очереди)
6. Запланированная дата начала (опц.)
7. Привязать дефекты из бэклога (опц.) — список открытых дефектов этой машины, можно отметить чекбоксами

`[✅ Создать]` → попадает в kanban в колонку «Создан» или «В работе» (если механик уже назначен).

### 4.5. Подраздел 2: Архив нарядов

Таблица всех завершённых нарядов с фильтром по периоду.

Колонки: дата, машина, механик, работ, факт время, эффективность, стоимость работ, запчасти, итого.

Тап на строку → детальный просмотр (read-only).

Экспорт CSV.

### 4.6. Подраздел 3: Справочник работ

Таблица `work_catalog`.

Колонки:

- Код
- Название
- Применимо к (типы машин)
- Норма времени (мин)
- Цена для клиента (₽)
- Себестоимость (норма-час × ставка)
- **Реальная норма** (медиана `actual_minutes` по последним 10 закрытым выполнениям) — вычисляемое поле
- **Отклонение** = (норма − реальная) / норма × 100% — если стабильно > 20%, нужно пересмотреть норму

Кнопка `[+ Добавить работу]`. Кнопка `[✏️ Редактировать]` на строке.

**Особенность:** при изменении нормы или цены — старые наряды НЕ пересчитываются (там цены и нормы зафиксированы снимком).

### 4.7. Подраздел 4: Склад запчастей

**Заголовок:** «Заморожено в запчастях: 293 200 ₽» (сумма остатков × закупочная цена).

**Таблица позиций:**

- Артикул / название
- Категория (масло / фильтры / сцепление / и т.д.)
- Совместимость (типы машин)
- Остаток (шт/л/кг)
- Минимальный порог (если ниже — алерт)
- Закупочная цена (последняя)
- Цена для клиента (рекомендуемая)
- Заморожено (₽) = остаток × закупка

**Фильтры:** по категории, остатку (мало/нормально), по совместимости.

**Действия:**

- `[+ Приёмка от поставщика]` — форма: позиция (или новая), количество, закупочная цена, поставщик, документ (фото накладной). Создаёт `part_movements` со знаком +, обновляет остаток. Создаёт `transaction expense` (закупка), привязанную к поставщику. Если pending — попадает в кредиторку.
- `[✏️ Корректировка остатка]` — для инвентаризации. Поле «Фактический остаток» + причина. Создаёт `part_movements` со знаком разницы (+/−).
- `[🗑 Списать]` — порча, кража, ошибка. С причиной.

**Заявки на закупку** — отдельная вкладка внутри склада. Список заявок от механиков (`status = pending_purchase`). Кнопка `[Купить]` — открывает форму приёмки с предзаполнением.

### 4.8. Подраздел 5: Эффективность механиков

**KPI по каждому механику за период:**

- Часов работы (по `work_time_logs`)
- Нарядов закрыто
- Средняя эффективность (факт vs норма)
- Заработано (по утверждённым нарядам)
- Выработка / час
- Простои (паузы по причинам)

**Карточка механика:**

```
👨‍🔧 Вадик
─────────────
Часов: 142    Нарядов: 18    Эффект: 91%
Заработал: 35 800 ₽    Выдано авансов: 20 000 ₽    К доплате: 15 800 ₽
─────────────
Простои: Обед 28ч | Жду запчасть 12ч | Конец смены 8ч | Другое 4ч

[📊 Подробно] [💰 Выплатить аванс]
```

**`[📊 Подробно]`** — детальный экран:

- График часов по дням
- Разбивка по типам работ (что чаще делал)
- Эффективность по работам (где медленнее нормы)
- Все наряды списком

### 4.9. Подраздел 6: Бэклог дефектов

Сводный список всех `defect_log` со статусом `open`, по всем машинам.

Фильтры: по машине, по узлу, по срочности.

Колонки: машина, узел, описание, срочность, фото (превью), кто нашёл, когда, статус.

Действия:

- `[📋 Запланировать ремонт]` — выбор даты + (опц.) привязка к существующему наряду или создание нового.
- `[❌ Отклонить]` — с причиной.
- `[Связать с нарядом]` — если работа уже идёт.

## 5. ФИНАНСЫ

- 📊 Чистый поток: +480 000 ₽

- Контрагент

**Фильтры:**

- По периоду
- По направлению
- По категории
- По кошельку
- По контрагенту
- По BU
- По статусам lifecycle и settlement

**Действия по строке:**

### 5.4. Вкладка 2: Дебиторка

**Шапка:** «Нам должны: 1 290 000 ₽ · Просрочено 30+: 85 000 ⚠️»

**Таблица должников:**

Колонки:

- Контрагент
- Сумма долга (общая по всем pending транзакциям)
- Самый старый долг (дата)
- Дни просрочки
- Контакт (телефон, можно нажать → tel: link)
- Менеджер (кто привёл клиента, опц.)
- Статус: 🟢 ожидание (срок не истёк) / 🟡 близко (срок через 3 дня) / 🔴 просрочка

**Фильтры:** по статусу, по сроку, по сумме.

**Тап на строку →** список конкретных долговых транзакций этого клиента: дата, сумма, описание (рейс №X / счёт-фактура №Y), статус, действия.

**Действия:**

- `[💰 Принять оплату]` — форма (как в MiniApp админа, погашение долга): сколько пришло, на какой кошелёк, дата.
- `[📞 Связаться]` — линк на звонок.
- `[📧 Напомнить]` — отправляет шаблонное email/SMS (v2).
- `[+ Добавить долг]` — ручное создание долга (например, зафиксировал в WhatsApp договорённость, нужно учесть).

**Реестр напоминаний:**

- При просрочке X дней — автоматический алерт на главной.
- При 30+ дней — пометка в карточке клиента, ограничение на новые отгрузки в долг (запрет в MiniApp у водителя «выбрать долг» для этого клиента — выводится предупреждение).

### 5.5. Вкладка 3: Обязательства (мы должны)

Аналогично дебиторке, но в обратную сторону.

**Категории обязательств:**

- 🏦 Кредиты банкам
- 📦 Поставщики (запчасти, ГСМ — задолженность)
- 👥 ЗП к выплате (агрегируется из `payroll_periods`)
- 🏛 Налоги (ручной ввод предстоящих)
- 📃 Аренда

**Карточка кредита:**

- Банк, тип
- Сумма основного долга на сегодня
- Ставка
- Платёж в месяц
- Дата следующего платежа
- График платежей (генерируется или импортируется CSV)
- Кнопка `[💸 Погасить часть]` — открывает форму: сумма, источник денег. Создаёт `transaction expense`, уменьшает остаток долга.

**Алерты:**

- За 3 дня до платежа по кредиту — на Главной.
- За 5 дней до даты налога — на Главной.

**Кнопка `[+ Добавить обязательство]`** — типы: кредит / разовый долг / периодический (аренда).

### 5.6. Вкладка 4: Кошельки

Таблица всех кошельков с балансами, аналогичная разделу «Деньги» в MiniApp админа, но шире:

Колонки:

- Название
- Тип
- Юрлицо
- Владелец (для подотчётов)
- Баланс
- Последняя транзакция
- Активен (yes/no)

Тап на кошелёк → детальный экран:

- Текущий баланс
- График баланса по дням за период
- Все транзакции (как в основной вкладке, но фильтр уже выставлен)
- Кнопка `[🔄 Перевод с этого кошелька]`
- Кнопка `[✏️ Редактировать]` (название, статус, владелец)

**Особенности подотчётов:**

- Балансы подотчётов **только увеличиваются от наличных операций** (приходы от клиентов с MiniApp), уменьшаются от расходов рейса и от инкассации.
- Если у уволенного водителя остался долг по подотчёту (взял аванс наличными, не отработал) — отображается красным. Действие: `[Перевести в дебиторку как личный долг сотрудника]`.

### 5.7. P&L и Cash Flow отчёты

Внутри раздела «Транзакции» есть переключатель на режим отчёта:

**P&L (отчёт о прибылях и убытках):**

- Выручка по категориям
- Себестоимость прямая (ГСМ, ФОТ водителей, запчасти ремонтов)
- Валовая прибыль
- Операционные расходы (аренда, связь, амортизация, ФОТ управления)
- Операционная прибыль
- Налоги
- Чистая прибыль

С фильтром по периоду и по BU (можно посмотреть P&L по конкретному направлению — Логистика LCV / СТО / и т.д.).

**Cash Flow:**

- Операционный поток
- Инвестиционный (покупка/продажа машин и оборудования)
- Финансовый (кредиты, дивиденды)

Это для v2, но архитектурно закладывать сразу — категории расходов нужно сразу размечать по типам потока.

## 6. ПЕРСОНАЛ

### 6.1. Назначение

Полная картина по сотрудникам: кто работает, сколько часов, сколько заработал, какие у него ставки. Расчёт ЗП за период.

### 6.2. Структура раздела

3 вкладки:

1. **Сотрудники** — реестр
2. **Расчёт ЗП** — периоды и выплаты
3. **График работы** (v2) — табель

### 6.3. Вкладка 1: Сотрудники

**Группировка по ролям:**

- Водители (11)
- Грузчики (3, с пометкой «совмещают с водителем»)
- Механики (2)
- Управление (2 — ты и Владелец)

**Карточка сотрудника на сетке:**

- Аватар (опц.)
- ФИО
- Роль / должность
- Машина (для водителей)
- ЗП за месяц (накоплено)
- Часов за месяц (для механиков)
- Статус: ✅ активен / ⏸ в отпуске / ❌ уволен
- Цветной индикатор: зелёный (норм), жёлтый (проблемы — низкая выработка / много просрочек / мало часов)

**Фильтры:** по роли, по статусу, по машине.

### 6.4. Детальный экран сотрудника

**Закладка 1: Анкета**

- ФИО, дата рождения, телефон
- Дата устройства
- Документы (паспорт, права для водителей — категория, срок действия)
- Закреплённая машина (для водителей)
- Текущая ставка / правила ЗП
- Комментарии (свободный текст — особенности, договорённости)

**Закладка 2: Работа за период**

- Дней работало
- Часов работало (для механиков — из таймера; для водителей — суммарная длительность рейсов)
- Заказов / нарядов
- Выручка с него (для водителей)
- Заработано (driver_pay/loader_pay/механик)
- Заработок в час (производный)
- Графики: часы по дням, заработок по неделям

**Закладка 3: Платежи**

- История выплат (авансы и полные)
- Текущий долг компании сотруднику (заработал − выдано)
- Кнопка `[💰 Выдать аванс/выплату]`

**Закладка 4: История** (v2)

- Изменения ставок
- Переходы между машинами
- Замечания / поощрения

**Действия в шапке:**

- `[✏️ Редактировать]`
- `[⏸ В отпуск]`
- `[❌ Уволить]` — модалка: дата увольнения, причина, проверка нет ли долгов по подотчёту.

### 6.5. Вкладка 2: Расчёт ЗП

**Выбор периода:** по умолчанию текущий месяц; можно выбрать любой.

**Таблица по сотрудникам:**

- ФИО / роль
- Заработано (по утверждённым рейсам/нарядам)
- Выдано авансами
- Остаток к выплате
- Статус: 🟡 в расчёте / 🔵 утверждено / 🟢 выплачено

**Кнопка `[Закрыть период]`** — когда все рейсы и наряды за месяц утверждены, период «закрывается»: создаются `payroll_periods` записи по каждому, баланс «к выплате» переходит в кредиторку компании перед сотрудником.

**Действия по строке:**

- `[👁 Подробно]` — раскрывает детализацию: список рейсов/нарядов с суммами.
- `[💰 Выплатить]` — форма: сумма (по умолчанию остаток), источник (касса / р/с / на карту), дата. Создаёт `transaction expense`, категория `PAYROLL_*`, кошелёк-получатель — подотчёт сотрудника. Затем сотрудник может «снять» эти деньги (учётно — переход с подотчёта в его личную руку, но с точки зрения компании — деньги ушли).

**Особенность:** ЗП водителей и механиков считается из ФАКТА (`trip_orders.driver_pay`, `service_order_works.actual_minutes × ставка`), не по формулам. Формулы — только подсказки.

### 6.6. Вкладка 3: График работы (v2)

Табель в виде календарной сетки: строки — сотрудники, колонки — дни, ячейки — статус (работал / выходной / больничный / отпуск).

На MVP это можно не делать — статус «работал» вычисляется автоматически: есть ли утверждённый рейс / наряд в этот день. Отпуска и больничные — ручной ввод.

## 7. НАСТРОЙКИ

### 7.1. Назначение

Конфигурация системы. Правки сюда вносятся редко, но критично, чтобы доступ был только у owner и admin.

### 7.2. Структура

Подразделы:

1. **Юрлица** — список, создание, редактирование
2. **Направления (BU)** — справочник
3. **Типы активов** — справочник
4. **Категории доходов и расходов** — справочник
5. **Правила ЗП** — `payroll_rules` (подсказки)
6. **Регламенты ТО** — `maintenance_regulations`
7. **Контрагенты** — реестр клиентов и поставщиков
8. **Цели и план** — целевая прибыль в месяц, KPI
9. **Интеграции** — настройки Опти24, Wialon, банка
10. **Пользователи и роли** — управление доступом
11. **Бэкапы и экспорт** — выгрузка БД, восстановление
12. **Журнал аудита** — `audit_log`

### 7.3. Юрлица

Список + форма (название, тип IP/OOO/SE, ИНН, налоговый режим, реквизиты, статус).

При создании нового юрлица — автоматически создаётся пара кошельков (`bank_account`, `cash_register` под него).

Переключатель «Активное юрлицо» в top bar — если их несколько, можно фильтровать всю систему по выбранному.

### 7.4. Контрагенты (расширенная версия)

Помимо CRUD-таблицы:

- Карточка контрагента: история заказов, общая сумма, дебиторка, средний чек, frequent products/services.
- Кредитный лимит: если клиенту разрешено в долг до X ₽, при превышении — автоматический запрет на новые отгрузки в долг.
- Метки: VIP / Проблемный / Постоянный.

### 7.5. Правила ЗП

CRUD таблицы `payroll_rules`. Каждое правило: имя, тип (percent/per_km/fixed/hourly_split), значение, применимо к (asset_type, trip_type), описание для UI.

При изменении правила — старые рейсы НЕ пересчитываются. Подсказка для новых рейсов будет уже из обновлённого правила.

### 7.6. Регламенты ТО

CRUD таблицы `maintenance_regulations`. Поля: тип актива, вид работы, интервал в км или в месяцах, шаблон работы (привязка к `work_catalog`).

При сохранении нового регламента или изменении — пересчитываются `maintenance_alerts` для всех машин этого типа.

### 7.7. Интеграции

**Опти24:**

- API ключ
- Связь карт с машинами (топливная карта №X → Газель 866)
- Кнопка `[🔄 Синхронизировать сейчас]`
- Лог синхронизаций (последние 20)
- Расписание pg_cron (когда настроим автоматику)

**Wialon:**

- Логин/токен
- Связь объектов с машинами
- Кнопка `[🔄 Тест соединения]`

**Банк:**

- Банк, БИК
- Загрузка выписки CSV: `[📁 Загрузить файл]` → парсинг → preview → подтверждение → создание транзакций.
- Лог импортов.

### 7.8. Пользователи и роли

CRUD таблицы `users`.

- При создании пользователя нужно ввести `max_user_id` (получаешь от сотрудника после первого захода в бот) или сгенерировать пригласительную ссылку.
- Назначение ролей (одна или несколько).
- Деактивация (статус `is_active = false`) — пользователь больше не может зайти, но история сохраняется.

### 7.9. Журнал аудита

Read-only список `audit_log`. Колонки: дата/время, кто, что (тип действия), сущность, до/после (diff). Фильтры по пользователю, по типу действия, по дате.

Это — последний рубеж при разборе «кто это сделал».

## 9. УВЕДОМЛЕНИЯ (колокольчик)

Иконка в top bar с бейджем количества непрочитанных.

Источники уведомлений:

- Завершён рейс (для admin/owner)
- Завершён наряд

## 10. ОТВЕТСТВЕННОСТЬ И ПРИНЦИПЫ ДИЗАЙНА

### 10.1. Что WebApp делает (а MiniApp — нет)

- Все апрувы (рейсы, наряды, транзакции в draft)
- Аналитика и отчёты
- CRUD справочников
- Расчёт ЗП за период
- Управление пользователями
- Интеграции (настройка)
- Журнал аудита
- Сравнение машин и сотрудников
- Графики и тренды

### 10.2. Что WebApp НЕ делает (это в MiniApp)

- Создание рейсов / заказов / нарядов от лица водителя/механика
- Запуск таймеров работ
- Полевые операции (приём наличных у водителя — это в MiniApp админа)

В WebApp эти действия **возможны теоретически** для админа (если он под ноутом и водитель приехал) — но это не основной сценарий. Лучше использовать MiniApp.

### 10.3. Дизайн-принципы

- **Плотность данных vs читаемость** — на dashboard много цифр, но не выше 5-7 на экран без скролла. Графики — крупные, читаются с расстояния.
- **Цветовая семантика:** зелёный = хорошо (прибыль, утверждено, оплачено), красный = плохо (убыток, просрочка, отменено), жёлтый = внимание (на ревью, аномалия), серый = нейтрально (черновик, в ожидании).
- **Никаких модалок-катастроф** — все диалоги маленькие, по делу. Большие операции (продажа машины, расчёт ЗП за период) — отдельные страницы.
- **Sticky elements** — фильтр периода и шапка таблиц прилипают к верху при скролле, чтобы не терять контекст.
- **Skeleton-загрузка** — никогда пустой экран, всегда контурный плейсхолдер пока грузится.
- **Empty states** — везде, где нет данных, дружелюбный плейсхолдер с подсказкой «Нажмите [+] чтобы добавить первую машину».

## 12. ЧТО ОСТАВЛЕНО НА v2

Чтобы не раздувать MVP:

- График работы / табель
- Система целей и KPI с трекингом по сотрудникам
- Email/SMS-напоминания клиентам по дебиторке
- Cash Flow отчёт (полный, не упрощённый)
- Бюджет и план vs факт
- CRM-функционал (история коммуникаций с клиентом)
- Мобильная адаптация WebApp (на MVP — только desktop ≥ 1200 px)
- Тёмная тема
- Многоязычность (на MVP — только русский)
- Подписи документов (счета, акты) — генерация PDF
- Интеграция с 1С / Контур.Эльба
- Уведомления в MAX-чат админам/владельцу о критических событиях
- Прогнозирование (ML на основе истории)
