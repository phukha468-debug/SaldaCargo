# 🛠 ЗАДАЧА 31: Очистка резервного входа и фикс редиректа диспетчера (Strict Auth Fix)

## 1. Контекст и Проблема
**Роль:** Senior Next.js / TypeScript Architect.
**Текущая проблема (Bug Report):** 1. Новые пользователи, заходящие через мессенджер МАХ, не видят экран со своим ID («Ваш аккаунт еще не активирован»). Вместо этого происходит ложный редирект на `/login`.
2. На странице `/login` пользователи видят ошибку «Пользователи не найдены». Это происходит из-за того, что там остался старый UI «Выбери Себя», который пытается запросить список пользователей из БД. RLS (Row Level Security) закономерно блокирует этот запрос для неавторизованных сессий.

**Цель:** Окончательно вычистить устаревший UI выбора ролей со страницы логина и расширить обработку ошибок в диспетчере для корректного перехвата незарегистрированных MAX-пользователей.

## 2. Жесткие ограничения (DO NOT TOUCH)
1. **НЕ ТРОГАЙ** роуты API (`/api/auth/max` и др.). Проблема только на клиенте.
2. **НЕ ТРОГАЙ** успешно работающий парсинг `WebAppData` в `app/page.tsx`.
3. **НЕ ДОБАВЛЯЙ** новые библиотеки или стейт-менеджеры.
4. **СОХРАНЯЙ** текущий UI-стиль (монохромный Zinc/Slate с акцентами Orange/Green).

## 3. Пошаговый план исправления (Выполнить строго по порядку):

### Шаг 1: Фикс перехвата ошибок в Root Dispatcher (`app/page.tsx`)
- Внутри `useEffect`, в логике `initAuth`, найди проверку ответа от `/api/auth/max`: `if (res.status === 403)`.
- **Проблема:** API может возвращать `404` или `401`, если пользователя нет в базе. Текущий код диспетчера пропускает эти статусы и уводит пользователя на `/login`.
- **Действие:** Замени `if (res.status === 403)` на более надежную проверку `if (!res.ok)`. 
- Любой неудачный ответ (когда пользователя нет в БД) должен приводить к вызову `setUnregisteredId(maxUserId)` и немедленной остановке выполнения функции через `return`.

### Шаг 2: Полная очистка страницы логина (`app/login/page.tsx`)
- Страница `/login` теперь является **только** fallback-сценарием для ручного ввода ПИН-кода.
- **Действие 1:** Полностью удали `useEffect` или функции, которые делают `fetch` списка всех пользователей из базы данных.
- **Действие 2:** Удали весь UI-код, отвечающий за рендеринг списка кнопок «Выбери Себя» (селектор пользователей).
- **Действие 3:** Оставь на странице **исключительно** форму ввода ПИН-кода (инпут и кнопка отправки).
- **Действие 4:** Убедись, что при успешном вводе ПИН-кода (ответ от `/api/auth/verify`) происходит жесткий `router.push('/')`, чтобы передать управление обратно корневому диспетчеру.

## 4. Ожидаемый результат
1. В коде `app/login/page.tsx` отсутствуют запросы к БД на получение списка сотрудников.
2. При переходе по прямой ссылке `/login` открывается только интерфейс ввода ПИН-кода без ошибок RLS.
3. При входе нового пользователя через МАХ, диспетчер ловит `!res.ok` и корректно показывает плашку с MAX ID, прерывая дальнейший редирект.
4. Проект проходит `pnpm build` без ошибок типизации.
