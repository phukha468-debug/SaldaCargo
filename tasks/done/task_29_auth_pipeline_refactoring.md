# 🛠 ЗАДАЧА 29: Рефакторинг конвейера маршрутизации и авторизации (Strict Blueprint)

## 1. Контекст и Проблема
**Роль:** Senior Next.js / TypeScript Architect.
**Текущая проблема (Bug Report):** Архитектура входа нарушена. Пользователи, заходящие через мессенджер МАХ, проходят двойную авторизацию (МАХ + запрос ПИН-кода). Возникает двойной экран выбора ролей. Редиректы перепутаны (кнопка Водителя ведет в Механика или на пустой экран).
**Цель:** Привести логику маршрутизации к строгому, линейному "Идеальному сценарию" без потери существующего рабочего кода парсинга и UI.

## 2. Жесткие ограничения (DO NOT TOUCH)
1. **НЕ ТРОГАЙ** логику парсинга `window.location.hash` (`WebAppData`) в `app/page.tsx`. Она работает идеально.
2. **НЕ ТРОГАЙ** UI-компонент экрана «Ваш аккаунт еще не активирован» (показ ID).
3. **НЕ ТРОГАЙ** UI-компонент кнопок «Выберите роль» (рендеринг массива `availableRoutes`).
4. **НЕ ТРОГАЙ** обход RLS (Service Role) в `/api/auth/max`.

## 3. Идеальный сценарий (Blueprint) для реализации:
- **МАХ-вход:** Читаем ID -> Идем в API -> Если ок, ставим куки сессии -> Показываем `RootDispatcher` (без захода на `/login`).
- **PIN-вход (Fallback):** Ввод ПИН-кода на `/login` -> Если ок, ставим куки -> Редирект на `/` (в `RootDispatcher`).
- **RootDispatcher (`/`):** Читает роли из сессии. Если роль одна -> моментальный `router.push(route)`. Если ролей несколько -> показываем UI кнопок.
