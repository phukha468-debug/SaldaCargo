# 🛠 ЗАДАЧА 33: Разблокировка Middleware и финализация Selector Flow

## 1. Контекст и Проблема
**Проблема:** Пользователь не видит изменений из TASK_32. Приложение по-прежнему принудительно открывает старую страницу `/login` с вводом ПИН-кода.
**Причина:** Текущий `middleware.ts` настроен на жесткую защиту: он редиректит на `/login` любого неавторизованного пользователя, включая тех, кто пытается попасть на главную страницу `/`, где теперь находится наш селектор.

## 2. Жесткие ограничения (DO NOT TOUCH)
1. **НЕ ТРОГАЙ** логику выбора ролей и машин в `app/page.tsx`.
2. **НЕ ТРОГАЙ** API `/api/users/public`.

## 3. Пошаговый план исправления (Выполнить строго по порядку)

### Шаг 1: Исправление `middleware.ts`
Необходимо разрешить доступ к корневому роуту (`/`), чтобы пользователь мог увидеть селектор ролей без активной сессии.
- Измени условие проверки редиректа. 
- **Было:** `if (!userId && !isAuthPage && !isAuthApi)`
- **Стало:** `if (!userId && !isAuthPage && !isAuthApi && pathname !== '/')`
- Это критическое изменение: теперь Middleware не будет выбрасывать пользователя на `/login`, если тот находится на главной.

### Шаг 2: Принудительный сброс (Force Clean)
Для устранения возможных конфликтов кэша Next.js и старых сессий:
1. Полностью удали папку `.next` в корне `apps/miniapp`.
2. Выполни `pnpm build`.
3. Убедись, что в логах сборки нет ошибок.

### Шаг 3: Проверка UI-связок
Убедись, что:
- Кнопка "Войти" (после выбора пользователя/машины) действительно устанавливает куку `salda_user_id`.
- После установки куки Middleware корректно пускает в защищенные разделы (`/driver`, `/mechanic`, `/admin`).

## 4. Ожидаемый результат
- При открытии приложения (путь `/`) Middleware больше не делает редирект.
- Сразу открывается экран выбора роли: **Водитель, Механик, Админ**.
- Процесс входа занимает 3 клика без ручного ввода текста.
