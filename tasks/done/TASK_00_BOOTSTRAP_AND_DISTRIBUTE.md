# TASK 00: Bootstrap и раскладка файлов по папкам

**Адресат:** Claude Code (или Gemini CLI).
**Длительность:** 5–10 минут.
**Зависимости:** ничего (это первое задание).
**Что должно быть готово до старта:** в `C:\salda` лежат файлы из манифеста (плюс часть, которой ещё нет — это ок).

---

## Цель

Разложить все файлы из корня `C:\salda` по правильным папкам внутри `docs/`, как описано в `00_BOOTSTRAP_MANIFEST.md`. Создать пустые заглушки для модулей и runbook. **Никакого кода, никаких пакетов, никаких миграций — только файлы и папки.**

---

## Алгоритм выполнения

### Шаг 1. Прочитать манифест

Открой и внимательно прочитай файл `00_BOOTSTRAP_MANIFEST.md` в корне `C:\salda`. Этот файл — твоя карта.

### Шаг 2. Проверить наличие файлов в корне

Перечисли файлы, которые **есть** в корне `C:\salda`. Сделай это командой `ls` или `dir` (в зависимости от ОС). Не выходи в подпапки — только корень.

Сверь со списком из манифеста. Те файлы, которых нет — пометь себе, что для них раскладка пропускается.

### Шаг 3. Создать структуру папок документации

Если их ещё нет, создай папки:

```
docs/
docs/architecture/
docs/business/
docs/business/examples/
docs/design/
docs/design/driver/
docs/design/mechanic/
docs/design/admin-mini/
docs/design/admin-web/
docs/flows/
docs/modules/
docs/modules/shared/
docs/modules/identity/
docs/modules/fleet/
docs/modules/finance/
docs/modules/logistics/
docs/modules/service/
docs/modules/payroll/
docs/modules/receivables/
docs/database/
docs/decisions/
docs/runbook/
```

**ВАЖНО:** не создавай папки `apps/`, `packages/`, `supabase/`, `scripts/`, `tests/`, `.github/` — они появятся в следующих заданиях.

### Шаг 4. Переместить и переименовать файлы

Для каждого файла из таблицы соответствия в манифесте:

- Если файл существует в корне — переместить (не копировать) в указанную папку с указанным новым именем.
- Если не существует — пропустить, отметить в отчёте.

**Используй команду `mv` или `Move-Item`**, не копирование.

Файлы, которые остаются в корне (не трогать):

- `START_HERE.md`
- `00_BOOTSTRAP_MANIFEST.md`
- `TASK_*.md` (все задания)

После раскладки удалить из корня:

- Не должны остаться файлы с префиксами `01_`, `02_`, ..., `28_`, `DESIGN_`, `EXAMPLE_` (они переименованы и перемещены).

### Шаг 5. Создать заглушки для модулей и runbook

Создай файлы со следующим содержимым:

**`docs/modules/shared/README.md`:**

```markdown
# Module: shared

**Статус:** TODO — заполнится при реализации модуля.

Назначение модуля и публичный API см. в `docs/architecture/02-modules.md`.
```

Аналогичные README создай для:

- `docs/modules/identity/README.md`
- `docs/modules/fleet/README.md`
- `docs/modules/finance/README.md`
- `docs/modules/logistics/README.md`
- `docs/modules/service/README.md`
- `docs/modules/payroll/README.md`
- `docs/modules/receivables/README.md`

(Меняя только имя модуля в первой строке.)

**`docs/database/README.md`:**

```markdown
# Database

**Статус:** TODO. Здесь будет описание схемы БД. Актуализируется при каждой миграции в `supabase/migrations/`.
```

**`docs/runbook/README.md`:**

```markdown
# Runbook

**Статус:** TODO. Здесь будет операционная информация для поддержки системы:

- как откатить деплой
- как восстановить из бэкапа
- что делать при падении интеграций
- контакты администраторов сервисов (Supabase, Vercel, МАХ)
```

**`docs/decisions/README.md`** — см. секцию «ADR README» в манифесте `00_BOOTSTRAP_MANIFEST.md`. Скопируй содержимое оттуда.

### Шаг 6. Проверка результата

После раскладки выполни проверки:

1. Запусти `ls docs/architecture/` — должны быть `01-overview.md`, `02-modules.md`, `03-conventions.md` (если все три файла были в корне).
2. `ls docs/business/` — должны быть все `*.md` файлы из манифеста (те, что были).
3. `ls docs/design/` — `README.md`, `tokens.md` и подпапки `driver/`, `mechanic/`, `admin-mini/`, `admin-web/`.
4. `ls docs/business/examples/` — все `*.jfif` и `*.jpg` файлы (если были).
5. `ls` корня `C:\salda` — должны остаться только: `START_HERE.md`, `00_BOOTSTRAP_MANIFEST.md`, и все `TASK_*.md`.

### Шаг 7. Обработка особых случаев

**Случай 1:** в корне есть файл, которого нет в манифесте.

Не трогать. В отчёте отметить как «неизвестный файл, оставлен в корне». Возможно, это новый файл из будущей сессии — пользователь разберётся.

**Случай 2:** файл из манифеста уже на правильном месте (повторный запуск).

Пропустить, в отчёте отметить «уже на месте».

**Случай 3:** в корне есть `12_business-service-catalog.md`, но папки `docs/business/` ещё нет.

Создай папку, потом перемести файл.

**Случай 4:** в корне нет файла из манифеста (ещё не пришёл).

Просто пропустить. В отчёте указать, что файл не найден. Не создавай заглушку для отсутствующего файла.

После выполнения задания всё нижеперечисленное должно быть истиной. Проверь каждый пункт:

- ❌ Не запускай `pnpm install`.
- ❌ Не создавай `package.json`, `tsconfig.json` или другие конфиги.
- ❌ Не модифицируй содержимое перемещаемых файлов (только имя, не содержимое).
- ❌ Не удаляй неизвестные файлы из корня.
- ❌ Не создавай ничего, что не указано в манифесте или этом задании.

## Если что-то идёт не так

- **Manifest не найден:** остановись, сообщи пользователю «не вижу файла 00_BOOTSTRAP_MANIFEST.md в корне C:\salda, без него я не могу выполнить задание».
- **Конфликт имён** (например, `docs/business/people.md` уже существует и не пуст): не перезаписывай, сохрани новый файл с суффиксом `.new` и сообщи пользователю в отчёте.
- **Нет прав на запись:** сообщи пользователю «не могу записать в C:\salda, возможно нужны права администратора».

После завершения остановись и жди следующей команды от пользователя — он передаст тебе `TASK_01_INIT_MONOREPO.md`.
