## 1. Что за проект

SaldaCargo — ERP-система для транспортной компании (Верхняя Салда).  
Цель: в будущем продать как коробочное решение для других транспортных компаний.

**Два приложения:**

- `apps/miniapp` — Next.js 15, мобильный интерфейс для водителей/механиков/админа (мессенджер MAX)
- `apps/web` — Next.js 15, десктопный дашборд для владельца и админа

**Стек:** Next.js 15, React 19, TypeScript, Supabase (PostgreSQL), TanStack Query, Tailwind CSS 4  
**Auth:**

- MiniApp: кастомная cookie `salda_user_id` (UUID из `users`)
- WebApp: кастомная cookie `salda_auth_token` (UUID из `users`) — через `/login` (телефон + PIN)

**Деплой:** Vercel (miniapp и webapp — оба задеплоены, автодеплой из `main`)

## 3. Что сделано в сессии 19 (26.05.2026)

### Персонал `/staff`

- **Частичный зачёт аванса при выплате ЗП:** кнопки 0%/30%/50%/Весь + ручной ввод суммы зачёта. API `POST /api/staff/settle` принимает `partial_offset`.
- **История транзакций сотрудника:** кнопка «История» в профиле → модалка со всеми PAYROLL + ADVANCE транзакциями. Редактирование суммы (PATCH) и удаление (DELETE) через `/api/staff/transactions/[txId]`.
- **Количество смен:** в карточке сотрудника показывается реальное число начислений ЗП за выбранный месяц (было `—`).

### Дебиторка `/finance` (ReceivablesPanel)

- **Привязка к контрагенту:** кнопка «Привязать/Изменить» под спойлером у каждого должника. При привязке — `PUT /api/receivables/[id]` для всех trip_order записей. Исправлен баг в API: `counterparty_id` теперь имеет приоритет над `description` при группировке.

### Гараж `/garage`

- **Электрики и сварщики в списке исполнителей:** `users.roles` — enum `user_role[]`. API `/api/users?role=mechanic` теперь возвращает роли `mechanic | mechanic_lead | welder | electrician` через `.filter('roles', 'ov', '{mechanic,mechanic_lead,welder,electrician}')`. То же исправление в MiniApp `/api/admin/mechanics`.
- **Исполнитель при закрытии наряда:** кнопка «Завершить наряд» теперь сохраняет выбранного исполнителя вместе с утверждением (раньше нужно было отдельно нажимать «Сохранить»).
- **Колонка «Исполнитель»** (было «Механик(и)»): переименована везде.
- **Информативность списка нарядов:** добавлены рег. номер авто, колонка «Тип/Дата» с бейджем Свой/Клиент и датой. Для клиентских машин показывается марка + госномер + имя клиента.
- **Утверждение наряда через ApproveOrderModal (MiniApp и WebApp):** добавлен выбор исполнителя прямо в форме утверждения.

### Роли в БД

- `user_role` enum содержит: `owner, admin, driver, loader, mechanic, mechanic_lead, accountant, welder, painter, electrician, handyman`
- Артём (электрик) имеет роль `electrician` в `users.roles`
- Макс (сварщик) имеет роль `welder` в `users.roles`

## 4. Архитектура учёта нарядов

### Жизненный цикл наряда

```
Создание → lifecycle_status: draft, status: created
Механик выполняет работу → status: completed, salary_paid: true
  → AUTO: транзакция PAYROLL_MECHANIC (settlement: pending = долг)
Все работы completed → Кнопка «Завершить наряд»
  → lifecycle_status: approved → Архив
```

### Расчёт ЗП за работу

```
salary = (факт.мин / 60) × нормачас × mechanic_salary_pct / 100
Если 2 исполнителя: каждый = salary / 2
settlement_status: 'pending' = долг (НЕ выплата)
```

### Нормачас

- Клиентский автомобиль: `sto_settings.hourly_rate` (2000 ₽/ч)
- Собственный автомобиль: `sto_settings.hourly_rate_own` (1600 ₽/ч)

## 6. Ключевые технические факты

- `users.roles` — enum `user_role[]`: `.contains('roles', ['driver'])`, НЕ `.eq('role', ...)`
- Для OR по массиву ролей: `.filter('roles', 'ov', '{mechanic,welder,electrician}')` (overlap `&&`)
- Все API-роуты: `createAdminClient()` + `(supabase.from('table') as any)`
- Деньги — только строкой: `"5000.00"`, компонент `<Money amount="5000.00" />`
- `crypto.randomUUID()` для idempotency_key
- **WebApp auth:** `salda_auth_token` cookie — читать через `requireAuth()` из `lib/auth.ts`
- **NULL в PostgreSQL:** `.neq('col', 'x')` не находит строки с `NULL`. Использовать `.or('col.is.null,col.neq.x')`
- Скроллируемый flex-child: обязательно `min-h-0` + `overflow-y-auto`
- `group-hover` на дочернем элементе: родительский `<tr>` / `<div>` ОБЯЗАН иметь класс `group`

## 7. Персонал — мультироль

### Дедупликация в /staff WebApp

Приоритет вкладок: `mechanic_lead` > `mechanic` > `driver` > `loader` > `office`.  
Реализовано через `assignedIds: Set` в `/api/staff/payroll/route.ts`.

### Редактирование истории транзакций сотрудника

- Разрешено: PAYROLL_DRIVER, PAYROLL_LOADER, PAYROLL_MECHANIC, ADVANCE_PAYMENT
- Pending PAYROLL — безопасно редактировать/удалять
- Completed PAYROLL и Авансы — только для исправления ошибки ввода (влияет на P&L)
- Зачёт аванса (income ADVANCE) — не трогать вручную
