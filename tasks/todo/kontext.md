# 🔄 АКТУАЛЬНЫЙ КОНТЕКСТ ПРОЕКТА: SaldaCargo (03.05.2026)

## 1. Суть проекта и Роль
**Проект:** SaldaCargo — комплексная система управления транспортной логистикой (автопарк, СТО, финансы).
**Стек:** Next.js 15 (App Router), TypeScript, Supabase (PostgreSQL + RLS), мессенджер MAX (для MiniApp).
**Стадия:** Завершена критическая фаза настройки авторизации и маршрутизации. Система переходит от "заглушек" к полноценной бизнес-логике.

## 2. Что мы решали (Последние изменения)
Мы сфокусировались на **бесшовном входе** и **надежной сессии**:
1.  **Auth Pipeline (Задача 29):** Реорганизован конвейер входа. 
    *   Убрана лишняя роль-селекция при ПИН-входе. 
    *   Линейный путь: `Вход (MAX/PIN) -> RootDispatcher (/) -> Проверка ролей -> Редирект в интерфейс`.
2.  **RLS Bypass (Задача 26):** Настроен обход Row Level Security через `createAdminClient` для API-роутов авторизации. Теперь неавторизованные пользователи могут быть найдены в БД по `max_user_id` или `phone`.
3.  **Session Fix (Задача 27):** Реализован полноценный Logout. Кнопка "Выход" теперь чистит куки через API и делает жесткий сброс состояния приложения. Исправлено зависание UI (бесконечный лоадер) при ошибках входа.
4.  **Security:** Исправлены проблемы с Git-пушами (исключены системные файлы `.github/workflows`, требующие специальных прав).

## 3. Текущая ситуация и архитектура
*   **Диспетчер (`app/page.tsx`):** Ключевой узел. Он умеет парсить `WebAppData` из мессенджера MAX, выполнять автоматический вход и распределять пользователя по интерфейсам (`/driver`, `/mechanic`, `/admin`).
*   **Авторизация:** Используется гибридный метод. Для MVP сессия держится на куке `salda_user_id`, которая проверяется в `Middleware` и `RootDispatcher`.
*   **База данных:** Пользователи в таблице `users` имеют поле `max_user_id` (для авто-входа) и `roles` (массив строк).
*   **Состояние UI:** Интерфейсы разделены на три зоны. Все они имеют базовые страницы и готовы к наполнению функционалом.

## 4. К чему идём (Ближайшие цели)
1.  **Наполнение функционалом Водителя:** Реализация полного цикла рейса (создание, заправки, расходы, завершение).
2.  **Модуль Механика:** Оживление работы с нарядами, таймерами ремонта и складским учетом.
3.  **Фин-учет:** Связка транзакций с рейсами и нарядами, расчет ЗП.
4.  **Аналитика для Админа:** Дашборды по юнит-экономике каждой машины.

## 5. Важные инструкции для AI
*   **База знаний:** Всегда начинай с `kb/SCHEMA.md` и `kb/meta/index.md`.
*   **Код:** Весь бизнес-функционал пишется в `packages/domain`. API-роуты в `apps/miniapp` должны быть максимально тонкими.
*   **БД:** Всегда помни про RLS. Если запрос из API к БД возвращает пустоту или 403 при наличии данных — проверь, используется ли `adminClient` там, где это уместно (только для системных действий).
*   **Стиль:** Монохромный, строгий дизайн (Zinc/Slate) с яркими акцентами (Orange/Green).
