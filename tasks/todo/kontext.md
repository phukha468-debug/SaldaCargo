## 1. Что за проект

SaldaCargo — ERP-система для транспортной компании (Верхняя Салда).  
Цель: в будущем продать как коробочное решение для других транспортных компаний.

**Два приложения:**

- `apps/miniapp` — Next.js 15, мобильный интерфейс для водителей/механиков/админа (мессенджер MAX)
- `apps/web` — Next.js 15, десктопный дашборд для владельца и админа

**Стек:** Next.js 15, React 19, TypeScript, Supabase (PostgreSQL), TanStack Query, Tailwind CSS 4  
**Auth:** кастомная cookie-схема (`salda_user_id`), НЕ Supabase Auth  
**Деплой:** Vercel (miniapp и webapp — оба задеплоены, автодеплой из `main`) — **всегда пушить после коммита**

## 3b. Что сделано в сессии 11 (16.05.2026)

### Раздел /counterparties — рефакторинг и аналитика

**Структура:**

- Убраны фильтры (Активные, С долгом, Спящие, Все)
- Два таба: **Постоянные** (список + карточка клиента) и **Разовые** (агрегированная аналитика)
- Кнопка "Добавить клиента" перенесена в строку с табами
- Убран заголовок "Контрагенты" сверху

**Карточка клиента:**

- 4 KPI-плитки: Выручка всего / Чистая прибыль / Рейсов / Средний чек
- Breakdown строка: Выручка − ЗП+ГСМ − Накладные X% = ЧП
- Барчарт выручки по месяцам (высота `h-20`)
- История рейсов: дата / водитель / машина / выручка / ЗП+ГСМ / ЧП за рейс

**Формула чистой прибыли:**

```
ЧП = Выручка − ЗП_водителя − ЗП_грузчика − ГСМ_пропорциональный − Выручка × overhead_pct
overhead_pct = Σ(REPAIR_PARTS + REPAIR_EXTERNAL + TAX + INSURANCE) / Σ(выручка компании)
```

**API:** новый эндпоинт `GET /api/counterparties/[id]/trips` — история рейсов клиента

**Разовые клиенты:** агрегированная статистика + сравнительная таблица Постоянные vs Разовые + доли выручки/прибыли

## 5. Архитектура учёта ЗП (важно!)

| Источник                                        | Что даёт                 | Где используется            |
| ----------------------------------------------- | ------------------------ | --------------------------- |
| `trip_orders.driver_pay / loader_pay`           | Начисленная ЗП из рейсов | P&L, "заработано" на /staff |
| `transactions` (category PAYROLL\_\* + ADVANCE) | Факт выплаты денег       | "выплачено" на /staff       |

**PAYROLL-транзакции входят в Finance расходы** — это правильно, они реальные расходы.  
**Расчёт долга в Staff:** `debt = MAX(earned - paid, 0)` где `paid` = PAYROLL + ADVANCE.

**Транзакции "закрытие подотчёта" НЕ должны вводиться через форму "Добавить расход".**  
Правильный путь: кнопка "ЗП сотрудников" → автоматически устанавливает `related_user_id`.

## 7. Дебиторка — архитектура

- **Источник долга из рейсов:** `trip_orders` где `settlement_status='pending'` AND `lifecycle_status='approved'`
- **Исторические долги:** `manual_receivables` где `settled=false`
- **"физ лицо" с описанием** → выносится в отдельную строку должника
- **Погашение trip_order:** PATCH `/api/receivables/[orderId]` → `settlement_status='completed'` + INSERT income-транзакция
- **Погашение manual:** PATCH `/api/receivables/manual/[id]` → `settled=true` + INSERT income-транзакция

---

## 8. Долги поставщиков — архитектура

- Долг = SUM(expense, pending, approved) - SUM(expense, completed, approved) по `counterparty_id`
- Дерябин ГСМ — `fuel_card` расходы из `trip_expenses` + ручные pending транзакции − платежи
- Pending expense-транзакции НЕ показываются в журнале финансов — только completed

**ID поставщиков:**

```
20000000-0000-0000-0000-000000000001 — Дерябин ГСМ (autoAccrue: fuel_card → долг)
20000000-0000-0000-0000-000000000002 — Новиков А.В. Запчасти
20000000-0000-0000-0000-000000000003 — Ромашин Запчасти
```

## 9. P&L vs Балансы — что что считает

| Раздел                            | Источник выручки                                                                     | Источник расходов                                             |
| --------------------------------- | ------------------------------------------------------------------------------------ | ------------------------------------------------------------- |
| **P&L (/finance)**                | `trip_orders` (lifecycle=approved + settlement=completed + trips.lifecycle=approved) | `transactions` (expense, approved, completed)                 |
| **Контрагенты (/counterparties)** | то же самое (унифицировано с Finance)                                                | —                                                             |
| **Автопарк (/fleet)**             | `trip_orders.amount` (approved)                                                      | `trip_expenses` + `driver_pay/loader_pay` + `part_movements`  |
| **Балансы кошельков**             | `transactions` (income, approved, completed, to_wallet_id)                           | `transactions` (expense, approved, completed, from_wallet_id) |
| **Дебиторка**                     | `trip_orders` (settlement=pending) + `manual_receivables` (settled=false)            | —                                                             |

## 11. Ключевые технические факты

- `users.roles` — массив: `.contains('roles', ['driver'])`, НЕ `.eq('role', ...)`
- Все API-роуты: `createAdminClient()` + `(supabase.from('table') as any)`
- Деньги — только строкой: `"5000.00"`, компонент `<Money amount="5000.00" />`
- `crypto.randomUUID()` для idempotency_key в webapp (НЕ пакет `uuid`)
- `useSearchParams()` в Next.js 15 — обязательно в `<Suspense>`
- WebApp работает **без cookie** (auth отключена локально) → API не должны требовать cookie для базовых операций
- Скроллируемый flex-child: обязательно `min-h-0` + `overflow-y-auto`
- Bottom-sheet в MAX WebView: `position:fixed` на body + `paddingBottom: calc(env(safe-area-inset-bottom,0px) + 56px)`
- Двухшаговое удаление: boolean state + useRef таймер 4 сек авто-сброс
- Новые таблицы в Supabase требуют явного `GRANT ALL ON TABLE ... TO postgres, anon, authenticated, service_role`
- **Транзакции ЗП вносить ТОЛЬКО через кнопку "ЗП сотрудников"**, не через форму расходов — иначе `related_user_id = null`
- **Подотчёт водителя** — только `payment_method = 'cash'`. `card_driver` — корпоративная карта, не подотчёт.
