## 1. Что за проект

SaldaCargo — ERP-система для транспортной компании (Верхняя Салда).

**Два приложения:**

- `apps/miniapp` — Next.js 15, мобильный интерфейс для водителей/механиков/админа (мессенджер MAX)
- `apps/web` — Next.js 15, десктопный дашборд для владельца и админа

**Стек:** Next.js 15, React 19, TypeScript, Supabase (PostgreSQL), TanStack Query, Tailwind CSS 4
**Auth:** кастомная cookie `salda_user_id` (miniapp) / `salda_auth_token` (webapp)
**Деплой:** Vercel (оба приложения, автодеплой из `main`)

### C. Финансовая модель — упрощение

**КОШЕЛЬКИ (2 вместо 3):**

- 🏦 Расчётный счёт (`10000000-0000-0000-0000-000000000001`)
- 💵 Касса (Наличные) (`10000000-0000-0000-0000-000000000002`)
- Карта (`...000003`) убрана из UI везде (в БД остаётся для истории)

**СПОСОБЫ ОПЛАТЫ В ЗАКАЗАХ (в форме водителя):**
| Тип клиента | Кнопка | payment_method | settlement | Куда идут деньги |
|---|---|---|---|---|
| Физлицо | 💵 Наличные | `cash` | completed | Касса (при апруве рейса) |
| Физлицо | 📱 QR/Безнал | `qr` | completed | Р/С (при апруве рейса) |
| Физлицо | ⏳ Долг | `debt_cash` | pending | Дебиторка → Касса при погашении |
| Юрлицо | 🧾 Счёт/Долг | `debt_cash` | pending | Дебиторка → Р/С при погашении |

**АПРУВ РЕЙСА — авто-транзакции:**

- `cash` + `card_driver` → Касса (income)
- `qr` → Р/С (income) ← НОВОЕ, раньше не создавалось
- `debt_cash` → только дебиторка, транзакция не создаётся

**ПОГАШЕНИЕ ДЕБИТОРКИ:**

- Юрлицо (`is_legal_entity = true`) → Р/С
- Физлицо (`is_legal_entity = false`) → Касса

### D. Контрагенты — юр/физ + дубли

- Миграция: `is_legal_entity BOOLEAN DEFAULT false` в таблице `counterparties`
  (применена 01.06.2026)
- WebApp `/counterparties`: чекбокс ЮЛ в каждой строке, секции Юрлица/Физлица,
  баннер дублей с инструкцией по мержу
- Driver API: фильтр `type IN (client, both)` + дедупликация по имени
- Форма заказа водителя: сначала выбор Юрлицо/Физлицо → затем поиск клиента

### E. Гараж — баг задвоения ЗП при повторном апруве

**Баг:** При "открытии заново" наряда (return) транзакции ЗП не удалялись,
при повторном апруве — двоились.

**Фикс 1 (развёрнут):** При `action=return` / `lifecycle=returned` теперь:

1. Удаляются все `transactions WHERE service_order_id = id`
2. Сбрасывается `salary_paid = false` на `service_order_works`

**Фикс 2 (развёрнут):** Webapp добавляет `service_order_id` в payroll-транзакции
(раньше не было — поэтому Фикс 1 не работал для webapp-апрувов)

### Расчёт ЗП механика

```
salary = (факт.мин / 60) × нормачас × mechanic_salary_pct / 100
Если 2 исполнителя: нормочасы делятся пополам
settlement_status: 'pending' → выплачивается через /staff
```

### Нормачас СТО

- Клиентский авто: `sto_settings.hourly_rate` (2000 ₽/ч)
- Собственный авто: `sto_settings.hourly_rate_own` (1600 ₽/ч)

### Правила API

- `users.roles` — массив: `.contains('roles', ['driver'])`
- Все роуты: `createAdminClient()` + `(supabase.from('table') as any)`
- Деньги — только строкой: `"5000.00"`, компонент `<Money amount="5000.00" />`
- Payroll-транзакции наряда ОБЯЗАНЫ содержать `service_order_id` (иначе не чистятся при возврате)
