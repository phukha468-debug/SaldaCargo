# ERP Module Creator

Скилл для создания новых разделов ERP-системы через структурированное интервью и последующую генерацию полнофункционального React-приложения.

## Философия

Пользователь часто знает ЧТО ему нужно, но не может формализовать это в структуру данных и бизнес-логику. Задача Claude — провести глубокое интервью, выявить все сущности, связи, процессы и правила, а затем сгенерировать готовый к использованию модуль.

Общайся с пользователем на его языке. Не используй технический жаргон без необходимости. Когда пользователь говорит «заказ-наряд», не нужно объяснять ему что такое «документ» — просто работай с его терминологией.

---

## Процесс работы

### Фаза 1: Интервью (ОБЯЗАТЕЛЬНАЯ)

Интервью — ключевая часть процесса. Без него модуль будет неполным. Интервью проводится пошагово, каждый шаг — отдельное сообщение с вопросами. Не задавай все вопросы сразу — это перегружает пользователя.

#### Шаг 1: Общее понимание

Выясни «большую картину»:

- Что за бизнес-процесс нужно автоматизировать?
- Кто будет работать в системе (роли)?
- Какие основные объекты учёта?
- Есть ли уже какая-то система (Excel, 1C, бумага)?

Пример первого сообщения:

```
Отлично, давайте создадим раздел «Гараж»! Для начала мне нужно понять масштаб.

Расскажите в свободной форме:
1. Какие основные задачи должен решать этот раздел?
2. Кто будет им пользоваться? (диспетчер, механик, бухгалтер...)
3. Как сейчас ведётся учёт? (Excel, тетрадка, 1С, ничего)
```

#### Шаг 2: Сущности и справочники

На основе ответа определи и уточни основные сущности. Для каждой выясни:

- Какие поля/характеристики важны?
- Как сущности связаны друг с другом?
- Какие статусы/состояния бывают?

Предложи свой вариант структуры и спроси «Я всё правильно понял? Что добавить/убрать?»

Типичные категории сущностей:

- **Справочники** — относительно постоянные данные (автомобили, сотрудники, запчасти, контрагенты)
- **Документы** — фиксируют события/операции (заказ-наряд, приходная накладная, акт выполненных работ)
- **Регистры** — накопительные данные (остатки на складе, начисления зарплаты)

#### Шаг 3: Бизнес-процессы и документооборот

Выясни жизненный цикл ключевых документов:

- Кто создаёт? Кто утверждает?
- Какие статусы проходит документ?
- Что происходит при смене статуса (списание со склада, начисление зарплаты)?
- Какие проверки/валидации нужны?

#### Шаг 4: Расчёты и формулы

Если есть расчёты (зарплата, стоимость, итоги):

- По какой формуле считается?
- Какие компоненты входят в расчёт?
- Есть ли нормативы, тарифы, коэффициенты?
- Как часто пересчитывается?

#### Шаг 5: Отчёты и аналитика

- Какие отчёты нужны руководству?
- Какие показатели важны для ежедневной работы?
- Нужны ли графики/дашборды?

#### Шаг 6: Подтверждение модели

Перед генерацией представь полную модель данных в читаемом виде:

```
📋 МОДЕЛЬ РАЗДЕЛА «ГАРАЖ»

СПРАВОЧНИКИ:
├── Автомобили (марка, модель, госномер, год, VIN, пробег)
├── Механики (ФИО, разряд, ставка/час, специализация)
├── Запчасти (название, артикул, ед.измерения, мин.остаток)
└── Виды работ (название, нормо-часы, категория)

ДОКУМЕНТЫ:
├── Заказ-наряд (автомобиль, дата, механик, работы[], запчасти[], статус)
├── Приходная накладная (поставщик, дата, позиции[])
└── Акт списания (заказ-наряд, позиции[])

РАСЧЁТЫ:
├── Стоимость заказ-наряда = Σ(работа.нормо-часы × тариф) + Σ(запчасть.цена × кол-во)
└── Зарплата механика = Σ(его заказ-наряды за период → нормо-часы × ставка)

ОТЧЁТЫ:
├── Остатки на складе
├── Зарплатная ведомость за период
├── История обслуживания автомобиля
└── Загрузка механиков
```

Спроси: «Всё верно? Можно генерировать?»

### Фаза 2: Генерация модуля

После подтверждения модели — генерируй React-приложение (.jsx).

#### Архитектура приложения

Генерируй ОДИН файл .jsx со следующей структурой:

```
1. Константы и конфигурация модели данных
2. Начальные данные (демо-данные для каждой сущности)
3. Утилиты и функции расчётов
4. Компоненты таблиц и форм
5. Компоненты деталей/карточек
6. Компонент навигации (боковое меню)
7. Главный компонент с роутингом по разделам
```

#### Обязательные элементы модуля

Каждый модуль ОБЯЗАН содержать:

1. **Боковое меню навигации** — с группировкой: Справочники, Документы, Отчёты
2. **CRUD для каждой сущности** — таблица + форма создания/редактирования
3. **Связи между сущностями** — выпадающие списки для выбора связанных объектов
4. **Статусы документов** — визуальные бейджи + кнопки перехода
5. **Расчёты** — автоматический подсчёт при изменении данных
6. **Поиск и фильтрация** — по основным полям каждой таблицы
7. **Демо-данные** — реалистичные примеры для каждой сущности
8. **Отчёты** — сводные таблицы с фильтрами по периоду

#### Стилизация

Используй корпоративный стиль, подходящий для ERP:

- Приглушённая палитра (тёмная боковая панель, светлая рабочая область)
- Чёткая типографика, хорошая читаемость таблиц
- Иконки через Lucide React
- Компактные, но не тесные элементы управления
- Адаптивные таблицы с горизонтальным скроллом при необходимости

Используй Tailwind CSS (только стандартные утилиты).

#### Технический стек

- React с хуками (useState, useMemo, useCallback)
- Lucide React для иконок
- Tailwind CSS для стилей
- Все данные — в state (без внешних хранилищ)
- Форматирование чисел и дат — встроенные Intl API

### Фаза 3: Итерация

После генерации спроси пользователя:

- Всё ли на месте?
- Что нужно добавить или изменить?
- Правильно ли работают расчёты?

Будь готов к нескольким раундам доработки. Типичные доработки:

- Добавить новое поле в справочник
- Изменить формулу расчёта
- Добавить новый отчёт
- Поменять порядок статусов
- Добавить новую сущность

## Примеры разделов

Для ориентации — типичные ERP-разделы и их сущности:

**Гараж / Автосервис**: автомобили, механики, запчасти, заказ-наряды, склад, зарплата
**HR / Кадры**: сотрудники, должности, отделы, приказы, отпуска, табель, зарплата
**Склад**: товары, поставщики, приходные/расходные накладные, инвентаризация, остатки
**Продажи / CRM**: клиенты, сделки, товары/услуги, счета, оплаты, воронка
**Производство**: заказы, спецификации, материалы, операции, выпуск, брак
**Проекты**: проекты, задачи, исполнители, этапы, бюджет, таймшиты

---

## Чего НЕ делать

- Не генерируй модуль без интервью — результат всегда будет неполным
- Не задавай все вопросы одним сообщением — разбивай на шаги
- Не используй технический жаргон без необходимости
- Не делай слишком маленький модуль «для примера» — делай полноценный
- Не забывай про демо-данные — пустой модуль невозможно оценить
- Не используй localStorage или sessionStorage — только React state
