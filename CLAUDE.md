## Что это за проект

**SaldaCargo** — ERP-система управления транспортным бизнесом (ИП Нигамедьянов, г. Верхняя Салда).
Автопарк ~12 единиц, направления: грузоперевозки + СТО. Цель: полный цифровой учёт.

**Два приложения, одна база данных (Supabase):**

- `apps/miniapp` — мини-приложение в мессенджере МАХ. Инструмент полевых операций: водители, механики, администратор на телефоне.
- `apps/web` — веб-дашборд для владельца и администратора. Инструмент анализа и контроля на ПК.

**Деплой:** Vercel (miniapp автодеплой из `main`), webapp — локально и на Vercel.

Я (владелец репо) единственный разработчик и работаю с тобой через AI-инструменты.
**Документация — главный источник истины.** Читай её перед каждой задачей.

## Разделение ответственности: MiniApp vs WebApp

**Это ключевое архитектурное решение. Дублировать функции между приложениями ЗАПРЕЩЕНО.**

### MiniApp — инструмент операций (ввод данных)

Всё, что делается быстро, в движении, с телефона:

- Создать рейс, добавить заказ, зафиксировать расход
- Завершить рейс с одометром
- Провести инкассацию (собрать наличные с водителя)
- Добавить доход или расход компании
- Одобрить / вернуть рейс на ревью
- Просмотреть оперативный KPI за день

### WebApp — инструмент анализа (отображение данных)

Всё, что удобно смотреть на большом экране, в графиках и таблицах:

- P&L по месяцам, динамика выручки и расходов
- Журнал транзакций с фильтрами
- Дебиторка: кто, сколько, насколько просрочено
- Кредиты: тело, остаток, платёжный календарь
- Автопарк: состояние, одометры, ТО
- Персонал: подотчёт, ЗП, эффективность
- Ревью рейсов (удобнее на ПК — допустимо в обоих приложениях)

**Действие из MiniApp немедленно отражается в WebApp — одна БД.**
**В WebApp НЕ нужно дублировать формы ввода из MiniApp.**

## Главные правила

### 1. Авторизация — кастомная cookie-схема (НЕ Supabase Auth)

Cookie `salda_user_id` = UUID из таблицы `users`. Никакого Supabase JWT.
Все API-роуты используют `createAdminClient()` (service_role key) — обходят RLS.
Читать userId в роутах:

```typescript
import { cookies } from 'next/headers';
const cookieStore = await cookies();
const userId = cookieStore.get('salda_user_id')?.value;
if (!userId) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
```

### 2. Supabase: всегда Admin Client, `as any` на `.from()`

```typescript
import { createAdminClient } from '@/lib/supabase/admin';
const supabase = createAdminClient();

// ✅ Правильно
const { data } = await (supabase.from('trips') as any).select('*');

// ❌ Неправильно
const { data } = (await supabase.from('trips').select('*')) as any;
```

### 3. Двухосный статус транзакций

У каждой транзакции два независимых статуса:

- `lifecycle_status`: `draft → approved → cancelled` (кто утвердил)
- `settlement_status`: `pending → completed` (прошли ли деньги)

**Балансы и P&L считаются ТОЛЬКО по `lifecycle_status='approved' AND settlement_status='completed'`.**

Сотрудники создают `draft`. Администратор переводит в `approved`. Прямые транзакции администратора создаются сразу `approved + completed`.

### 4. Деньги — только строкой

```typescript
// ✅ Правильно — string с двумя знаками
const amount: string = "5000.00";
<Money amount={amount} />  // компонент из @saldacargo/ui

// ❌ Неправильно
const amount: number = 5000;
```

В БД — `DECIMAL(12,2)`. Никогда `float` или `number`.

### 5. ЗП — только ручной ввод

ЗП водителя (`trip_orders.driver_pay`) вводится вручную. Система показывает подсказку (~30%), но не пересчитывает.
**Не пиши автоматический пересчёт ЗП. Не пиши триггеры, которые правят введённую цифру.**

### 6. Soft-delete, не DELETE

Никаких `DELETE` для бизнес-данных. Все «удаления» — через `lifecycle_status='cancelled'` с `cancelled_reason`.
Исключение: `trip_expenses` — физическое удаление технически допустимо.

### 7. Идемпотентность на мутациях

Каждая форма создания генерирует `idempotency_key = crypto.randomUUID()` при монтировании.
Сервер при дубле (PostgreSQL 23505) возвращает существующую запись с 200.

### 8. `users.roles` — массив

```typescript
// ✅ Правильно
.contains('roles', ['driver'])

// ❌ Неправильно
.eq('role', 'driver')
```

### 9. Балансы не хранятся статически

Любой баланс вычисляется «на лету» из журнала транзакций. Никогда не храни баланс как поле таблицы.

### 10. Синхронизация MiniApp ↔ WebApp — ОБЯЗАТЕЛЬНО

**При любом изменении бизнес-логики в API одного приложения:**

1. Открой `kb/wiki/CrossApp_Features.md`
2. Найди строку с этим endpoint'ом
3. Открой зеркальный файл в другом приложении
4. Внеси аналогичное изменение
5. Оба файла — в одном коммите
6. Обнови статус в `CrossApp_Features.md`

**Критические инварианты (нарушать нельзя — расходятся данные):**

- Апрув рейса → `trips.lifecycle=approved` + `trip_orders.lifecycle=approved` **в обоих**
- Возврат рейса → `trips.lifecycle=returned` + `trip_orders.lifecycle=draft` + очистка одометра **в обоих**
- Погашение дебиторки → `settlement=completed` + INSERT income-транзакция **в обоих**
- Отменённые рейсы → фильтровать `neq lifecycle_status=cancelled` везде в списках водителя

## Когда создаёшь миграцию БД

1. Прочитай `kb/wiki/Справочник_таблиц_БД.md` — текущая схема.
2. Прочитай `kb/wiki/Миграции_БД.md` — регламент.
3. Создай файл в `supabase/migrations/<timestamp>_<описание>.sql`.
4. Имя файла описывает действие: `20260507000002_add_loans_table.sql`.
5. В начале файла укажи комментарием как откатить.
6. Обнови `kb/wiki/Справочник_таблиц_БД.md`.
7. Сгенерируй типы: `pnpm db:types`.
8. Закоммить миграцию + типы + документацию **одним коммитом**.

## Когда добавляешь функциональность

**Сначала определи приложение:** это операция (MiniApp) или аналитика (WebApp)?

Для API-роута:

1. Парсинг body через Zod-схему.
2. Запрос к Supabase через `createAdminClient()`.
3. Форматирование ответа.

Никакого SQL-кода, вынесенного в компоненты. Никакой бизнес-логики в клиентском коде.

## Система domain-модулей (`packages/domain/`)

Система существует и частично работает. Понимать её обязательно.

### Что это

9 пакетов с бизнес-логикой, разбитых по областям: `shared`, `identity`, `fleet`, `finance`, `logistics`, `service`, `payroll`, `receivables`, `integrations`. Каждый импортируется как `@saldacargo/domain-<имя>`. Правила импортов между ними описаны в `docs/architecture/02-modules.md`.

### Почему сейчас не используется полностью

MiniApp строился быстро: логика писалась прямо в API-роутах, это быстрее и проще для одного разработчика. Система была спроектирована заранее, но при написании кода выбирался путь наименьшего сопротивления. Это технический долг — не катастрофа, но нужно остановить его рост.

### Правило: когда класть в domain, когда в роут

**В API-роут (просто CRUD):**

```typescript
// Простой SELECT, INSERT, UPDATE — оставить в роуте
GET /api/trips → supabase.from('trips').select(...) → OK в роуте
```

**В domain-пакет (логика и вычисления):**

```typescript
// Расчёт, который нужен в нескольких местах — выносить в domain
// packages/domain/finance/src/index.ts
export function calculateDriverBalance(transactions, driverId): string { ... }

// packages/domain/receivables/src/index.ts
export function calculateOverdueDays(dueDate: string): number { ... }
```

**Три триггера для выноса в domain:**

1. Одна и та же логика нужна и в miniapp, и в webapp
2. Это вычисление (не SELECT), которое хочется покрыть тестом
3. Это бизнес-правило (не техника): «как считается баланс», «что считается просрочкой»

### Что уже в domain и используется

| Пакет     | Что внутри                                                | Импортируй как               |
| --------- | --------------------------------------------------------- | ---------------------------- |
| `shared`  | formatDate, formatMoney, addMoney, generateIdempotencyKey | `@saldacargo/shared`         |
| `finance` | calculateWalletBalance, getPaymentMethodLabel             | `@saldacargo/domain-finance` |
| `service` | getMechanicSummary, startWork, stopWork                   | `@saldacargo/domain-service` |

### Что НЕ нужно делать

- Не переписывать существующие рабочие API-роуты ради архитектуры
- Не создавать domain-функцию для простого CRUD который используется в одном месте
- Не нарушать направление импортов (см. `docs/architecture/02-modules.md`)
