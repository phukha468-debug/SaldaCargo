## Что это за проект

**SaldaCargo** — система управления транспортным бизнесом в Верхней Салде. Состоит из двух приложений:

- **MiniApp** — мини-приложение в мессенджере МАХ для водителей, механиков и админа в полевых условиях.
- **WebApp (Dashboard)** — веб-дашборд для админа и владельца с полной аналитикой и управлением.

Бэкенд — Supabase (PostgreSQL + Auth + Storage + Realtime). Деплой на Vercel.

Я (владелец репо) единственный разработчик и работаю с тобой через AI-инструменты. Это значит: документация — главный источник истины, и ты обязан её читать перед каждой задачей.

## Работа с задачами

Все задачи на разработку и настройку находятся в папке `tasks/todo/`.

1. **Бери задачу** из `tasks/todo/` (обычно это файл с наименьшим порядковым номером, например `TASK_00_...`).
2. **Выполняй её** в точности по шагам и критериям приёмки.
3. **После завершения** подготовь отчёт для пользователя.
4. **Перемещение файла:** пользователь сам переместит файл в `tasks/done/` после проверки, либо может попросить тебя сделать это в рамках задачи.

## Главные правила

### 1. Граф модулей — священен

В `packages/domain/` девять модулей. Они организованы в 4 уровня (см. `02-modules.md`):

- Уровень 1 (фундамент): `shared`, `identity`
- Уровень 2 (ядро): `finance`, `fleet`
- Уровень 3 (бизнес): `logistics`, `service`, `payroll`, `receivables`, `integrations`
- Уровень 4 (оркестрация): API routes в `apps/web/app/api/` и `apps/miniapp/app/api/`

**Импорт может идти только сверху вниз.** Соседи на одном уровне не зовут друг друга.

Если кажется, что нужна связь между соседями — поднимаешь её на уровень 4 (use-case в API route склеивает оба модуля). Не лепи костыли.

### 2. API routes — тонкие обёртки

Каждый API route делает только три вещи:

1. Парсит body через Zod-схему.
2. Вызывает функцию из `@saldacargo/domain-<модуль>`.
3. Форматирует ответ.

Никакого SQL внутри API route. Никакой бизнес-логики. Если видишь толстый route — значит, его надо рефакторить.

### 3. Двухосный статус транзакций

У каждой транзакции **два независимых статуса**:

- `lifecycle_status`: `draft → approved → cancelled` (кто утвердил)
- `settlement_status`: `pending → completed` (прошли ли деньги)

**Балансы и P&L считаются ТОЛЬКО по `approved + completed`.** Это правило вшито в SQL-вьюхи и в логику `domain/finance`.

Сотрудники (driver, mechanic, loader) создают только `draft`. Внешние API (Опти24, банк) создают сразу `approved`.

### 4. ЗП — ручной ввод, не формула

ЗП водителя по каждому заказу вводится ВРУЧНУЮ в поле `trip_orders.driver_pay`. Система только показывает подсказку («~30%»), не пересчитывает.

ЗП механика — по факту времени из `work_time_logs`, помноженному на ставку. Норматив — для отчёта об эффективности, не для расчёта ЗП.

**Не пиши автоматический пересчёт ЗП.** Не пиши триггеры, которые «правят» введённую цифру. Это нарушит весь учёт.

### 5. Деньги — только DECIMAL

Никогда не используй `number` для денежных сумм в JS/TS. Только `string` (с двумя знаками после запятой) и арифметика через хелперы из `@saldacargo/domain-shared/money`. В БД — `DECIMAL(12,2)`.

### 6. Soft-delete, не DELETE

Никаких `DELETE` в SQL. Все «удаления» — через статус `cancelled` с обязательным `cancelled_reason`. История транзакций неизменна.

### 7. Идемпотентность на мутациях

Каждая мутация от клиента (создание заказа, расхода, наряда) приходит с `idempotency_key` (UUID v4). Сервер хранит ключи 24 часа в Redis или в отдельной таблице `idempotency_keys` и при дубле возвращает прошлый ответ. Без этого — двойные клики и плохая сеть будут портить данные.

### 8. RLS на всех таблицах

В Supabase везде включён Row Level Security. Водитель видит только свои рейсы, механик — только свои наряды. Politики живут в миграциях, не в коде приложения.

## Когда создаёшь миграцию БД

1. Прочитай текущую `docs/database/schema.md`.
2. Создай файл миграции в `supabase/migrations/<timestamp>_<имя>.sql`.
3. Имя миграции описывает что именно делает: `20250428143000_add_work_catalog_table.sql`.
4. Каждая миграция должна быть обратимой — комментарием в начале укажи, как откатить.
5. Обнови `docs/database/schema.md` — добавь описание новой таблицы или изменённых полей.
6. Обнови `docs/modules/<затронутый-модуль>/README.md` если меняются публичные сущности модуля.
7. Прогенерируй типы: `pnpm gen:types` (вызывает `supabase gen types typescript`).
8. Закоммить миграцию + типы + обновлённую документацию **одним коммитом**.

---

## Когда добавляешь новую функциональность

1. Определи, к какому модулю относится изменение (один или несколько).
2. Если функция кросс-модульная — её место в API route (use-case на уровне 4), не в самом модуле.
3. Напиши Zod-схему для входных данных в `domain/<модуль>/schemas.ts`.
4. Напиши функцию в `domain/<модуль>/services.ts`. Она возвращает `Result<T, E>` (см. `domain/shared/result`).
5. Экспортируй её через `index.ts` модуля.
6. Используй её в API route или из use-case.
7. Если логика нетривиальна (не CRUD) — добавь юнит-тест в `services.test.ts`.

## Когда принимаешь нетривиальное решение

Создай ADR (Architecture Decision Record) в `docs/decisions/`. Формат:

```markdown
# 0007. Использовать pg_cron вместо Vercel Cron

**Дата:** 2026-04-28
**Статус:** Принято

## Контекст

...

## Решение

...

## Последствия

...
```

Номера сквозные (0001, 0002, ...). После создания — упомяни ADR в коммите: `feat: переход на pg_cron (см. ADR-0007)`.

## Стек (короткая шпаргалка)

- **Frontend:** Next.js 15 (App Router), React 19, TypeScript strict, Tailwind, shadcn/ui, Tremor (графики)
- **Backend:** Next.js API Routes, Supabase JS Client (НЕ Prisma)
- **БД:** Supabase PostgreSQL + RLS + pg_cron
- **Auth:** Supabase Auth + MAX OAuth
- **Storage:** Supabase Storage (фото одометров, чеки, дефекты)
- **Realtime:** Supabase Realtime (обновления Главной, Ревью смены)
- **Формы:** react-hook-form + zod
- **Запросы:** TanStack Query
- **Тесты:** Vitest
- **Mono:** pnpm workspaces + Turbo (опционально)
- **Деплой:** Vercel Hobby + GitHub Actions
- **Ошибки:** Sentry Free
