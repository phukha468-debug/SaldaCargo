# SaldaBiz — Архитектура системы v2.3

**Продукт:** Mini App в мессенджере MAX + Web Dashboard
**Стек:** Next.js 15 + Supabase (PostgreSQL, pg_cron) + Vercel (Hobby / Free)
**Цель:** Оцифровка мульти-направленного бизнеса, выход на чистую прибыль 1 500 000 руб./мес.

---

## Changelog v2.3 (от v2.2)

| Изменение | Причина |

## 0. Принципы (без изменений)

1. Сотрудник = поставщик данных (draft).
2. Два статуса: lifecycle × settlement.
3. Источник истины = журнал транзакций.
4. Внешние API = auto-approved.
5. Удалений нет, только cancelled.
6. Всё измеримо деньгами.

## 1. Технологический стек

### 1.1. Стоимость MVP = 0 ₽/мес

| Компонент | Тариф | Стоимость |
|---|---|---|
| Supabase | Free (500 МБ БД, 1 ГБ Storage, 50k API/мес) | 0 |
| Vercel | Hobby (бесплатный деплой Next.js) | 0 |
| Домен saldabiz.ru | ~200 ₽/год | ~17 ₽/мес |
| Sentry | Free (5k ошибок/мес) | 0 |
| **Итого** | | **~17 ₽/мес** |

Переход на платные тарифы — только когда Free-лимиты закончатся (при ~50+ активных пользователях или >500 МБ данных). К тому моменту бизнес уже будет зарабатывать.

### 1.2. Фоновые задачи без Vercel Pro

**pg_cron** — встроенное расширение PostgreSQL в Supabase (доступно на Free). Позволяет запускать SQL-функции по расписанию прямо в базе.

| Задача | Расписание | Реализация |
|---|---|---|
| Синхронизация Опти24 | Пока: ручная кнопка «🔄 Синхронизировать» в Dashboard. Потом: pg_cron каждые 15 мин | Supabase Edge Function вызывается по кнопке или по cron |
| Амортизация | 1-е число: ручная кнопка «Начислить амортизацию» | SQL-функция, admin вызывает раз в месяц |
| Банковская выписка | Ручной импорт CSV | Загрузка файла через Dashboard |
| Проверка регламентов ТО | Триггер в БД при обновлении пробега | Автоматически, без cron |
| Уведомления в MAX | При событии (триггер) → webhook | Supabase Edge Function |

**Итого:** на MVP **ни одной платной подписки на cron**. Три кнопки в Dashboard вместо автоматики. Автоматику включим через pg_cron когда захотим — бесплатно.

## 2. Система ЗП — переработанная на основе реальных данных

### 2.1. Как реально работает ЗП (по путевым листам)

Анализ двух реальных путевых листов от 19.08.2024 показал:

**Машина №866 (Газель), Вова + Серёга:**
```
Интерьер    2700  →  Вод 1000 (37%)  Груз 1000 (37%)
Переезд     6000  →  Вод 2000 (33%)  Груз 2000 (33%)
Див         2000  →  Вод  650 (33%)  Груз  650 (33%)
Див         4000  →  Вод 1300 (33%)  Груз 1300 (33%)
Левша        650  →  Вод  200 (31%)  Груз    0
(б/н)        700  →  Вод  200 (29%)  Груз    0

Итого:     16050     Вод 5350         Груз 4950
ГСМ: 1000
Прибыль фирмы: 4750
```

**Машина №099 (Газель), Денис + Вова (грузчик):**
```
Грут        1300  →  Вод  500 (38%)  Груз  350 (27%)
Мистер       700  →  Вод  200 (29%)  Груз    0
(б/н)       1000  →  Вод  300 (30%)  Груз    0
(б/н)       1000  →  Вод  300 (30%)  Груз    0
Левша       1150  →  Вод  450 (39%)  Груз  250 (22%)  ← с грузом
650×5       3250  →  Вод 1000 (31%)  Груз    0
Переезд     1000  →  Вод  300 (30%)  Груз    0

Итого:      9400     Вод 3050         Груз  600
ГСМ: 1000
Прибыль фирмы: 4750
```

**Выводы:**
1. **«30%» — ориентир, не формула.** Реальный % зависит от типа работы, наличия грузчика, объёма заказа.
2. **На мелких заказах (650–700₽)** — фиксированная оплата 200₽ за точку (~30%).
3. **С грузчиком** — оба получают больше (33–38% каждый), потому что работа физически тяжелее.
4. **Переезды** — равные доли (33/33/34 или близко).
5. **ЗП за день считается как СУММА по всем строкам**, не как % от общей выручки.

### 2.2. Решение: ручной ввод + умная подсказка

**При добавлении заказа в рейс (Mini App) водитель вводит:**
1. Клиент (или «б/н» — без названия)
2. Сумма заказа
3. ЗП водителя (поле с подсказкой: «~300₽ (30%)» серым цветом)
4. ЗП грузчика (если есть грузчик в экипаже) — аналогично с подсказкой
5. Способ оплаты

**Подсказка** считается по `payroll_rules` для этого типа рейса/актива, но **не обязывает**. Водитель может ввести любую цифру. Admin при ревью видит отклонения от нормы.

**В Dashboard при ревью:** если ЗП по строке > 40% от суммы или < 25% → строка подсвечена жёлтым (предупреждение, не блокировка).

### 2.3. Payroll Rules — теперь как справочник подсказок

`payroll_rules` хранит не «формулы расчёта», а **рекомендованные ставки** для подсказок:

```
Газель + обычный заказ → подсказка 30%
Газель + с грузчиком → подсказка водитель 33%, грузчик 33%
Газель + переезд → подсказка водитель 33%, грузчик 33%
Газель + межгород → подсказка 25%
Валдай + локальный → подсказка 10₽/км + 1000₽/суточные
Валдай + почасовой → подсказка 30%
```

**Итого ЗП за смену** = SUM(all payroll entries per trip). Это число идёт в `payroll_periods` при закрытии месяца.

### 2.4. Переезды (уточнённые правила)

**Ставка:** 3 000 ₽/час
**Распределение при 4 часах (12 000₽):**
- Газель (фирма): 4 000₽
- Водитель: 4 000₽
- Грузчик: 4 000₽

В транзакциях:
```
income:  12 000₽ от клиента → wallet (нал/безнал)
expense:  4 000₽ ЗП водителя (category: PAYROLL_DRIVER)
expense:  4 000₽ ЗП грузчика (category: PAYROLL_LOADER)
→ Прибыль фирмы: 4 000₽ (считается в P&L автоматически)
```

### 2.5. Межгород (любая газель)

**Ставка:** 12 000₽ (фиксированная)
**Водителю:** 25% = 3 000₽

### 3.2. Амортизация по остаточной стоимости

```
Ежемесячная амортизация = остаточная_стоимость / оставшийся_срок_мес

Пример Валдай 2012:
  1 600 000 / 120 мес = 13 333₽/мес

Каждый месяц:
  current_book_value -= 13 333
  → transaction expense (category: DEPRECIATION_VEHICLE, amount: 13 333, auto-approved)
```

## 5. Как реальный путевой лист превращается в данные системы

### Пример: Машина №866, 19.08.2024

**Водитель Вова открывает Mini App, начинает рейс:**
```
→ INSERT trip: asset=866, driver=Вова, started_at=08:30, odometer_start=фото
→ Экипаж: грузчик=Серёга (выбирает из списка)
```

**В течение дня добавляет заказы (кнопка «+ Заказ»):**

| Действие в Mini App | Что в БД |
|---|---|
| Клиент: «Интерьер», Сумма: 2700, ЗП вод: 1000, ЗП груз: 1000, Оплата: Наличные | `income tx: 2700, from: ext_client, to: driver_wallet, draft+completed` + 2 записи payroll |
| Клиент: «Переезд», Сумма: 6000, ЗП вод: 2000, ЗП груз: 2000, Оплата: Наличные | то же |
| Клиент: «Див», Сумма: 2000, ЗП вод: 650, ЗП груз: 650, Оплата: Наличные | то же |
| Клиент: «Див», Сумма: 4000, ЗП вод: 1300, ЗП груз: 1300, Оплата: Наличные | то же |
| Клиент: «Левша», Сумма: 650, ЗП вод: 200, ЗП груз: 0, Оплата: Наличные | то же, без payroll грузчика |
| Клиент: «б/н», Сумма: 700, ЗП вод: 200, ЗП груз: 0, Оплата: **Долг** | `income tx: 700, from: ext_client, to: cash_office, draft+pending` |

**Расход: кнопка «+ Расход»:**
| Действие | БД |
|---|---|
| ГСМ: 1000₽, наличные | `expense tx: 1000, category: FUEL, from: driver_wallet, to: ext_supplier` |

**Завершение рейса:**
```
→ UPDATE trip: ended_at=18:00, odometer_end=фото, status=completed
→ UPDATE assets.odometer_current ⚡ триггер ТО
```

**Вечером ты открываешь Dashboard → «Ревью смены»:**
```
Машина 866, Вова:
  6 заказов: 16 050₽
  ЗП водителя: 5 350₽ (33.3% средний)
  ЗП грузчика: 4 950₽ (30.8% средний)
  ГСМ: 1 000₽
  Прибыль: 4 750₽
  ⚠️ 1 долг: 700₽ (б/н клиент)

  [✅ Подтвердить всё]  [✏️ Редактировать]
```

**После подтверждения:**
- Все транзакции → `lifecycle: approved`
- Подотчёт Вовы: +10 100₽ (наличные от клиентов минус ГСМ)
- Дебиторка: +700₽ (долг б/н клиента)
- P&L «Логистика LCV» за день: +4 750₽ (с машины 866)

## 8. Способы оплаты (уточнённые)

В Mini App при добавлении заказа — 5 вариантов:

| Кнопка | Что в БД | Баланс |
|---|---|---|
| 💵 **Наличные** | from: ext_client, to: **driver_wallet**, settled: completed | Подотчёт водителя ↑ |
| 📱 **QR на р/с** | from: ext_client, to: **ip_rs**, settled: completed | Расч. счёт ИП ↑ (приход мгновенный) |
| 🏦 **Безнал (счёт)** | from: ext_client, to: **ip_rs**, settled: **pending** | Дебиторка ↑ |
| ⏳ **Долг (нал)** | from: ext_client, to: **cash_office**, settled: **pending** | Дебиторка ↑ |
| 💳 **На карту водителя** | from: ext_client, to: **driver_wallet**, settled: completed | Подотчёт ↑ (= наличные по учёту) |

**QR vs Безнал:**
- QR = деньги пришли мгновенно → `completed`
- Безнал по счёту = клиент заплатит позже → `pending` (дебиторка)
