# Flow: Синхронизация с Опти24

## Контекст

Опти24 — сервис топливных карт. Каждая карта привязана к машине. Когда водитель заправляется — Опти24 фиксирует заправку (литры, сумма, АЗС).

Раз в N часов (или по кнопке) — мы дёргаем API Опти24 и подтягиваем новые заправки. Каждая заправка → транзакция расхода со статусом `auto-approved` (внешний API — доверенный источник).

---

## Шаги

### Шаг 1. Запуск

**MVP — вручную:**
WebApp → Настройки → Интеграции → Опти24 → `[🔄 Синхронизировать сейчас]`.

**Позже — pg_cron каждые 15 минут.**

```
POST /api/admin/integrations/opti24/sync
  Body: { idempotency_key }

Use-case:
  1. identity.requireRole(user, ['admin','owner'])
  2. integrations.opti24.sync({ executor_id: user.id })
     → внутри:
       a. Получаем last_sync_at из настроек интеграции
       b. Запрос к API Опти24:
          GET /transactions?since=<last_sync_at>
          Headers: Authorization: Bearer <opti24_api_key>
       c. Для каждой transaction в ответе:
          - Проверка: уже есть в БД? (по external_transaction_id)
            Если да — skip (идемпотентность)
          - Маппинг card_number → asset_id (через fuel_cards таблицу)
          - INSERT в fuel_transactions_raw (raw данные)
          - finance.createTransaction({
              direction: 'expense',
              amount: opti_tx.amount,
              category_id: FUEL,
              asset_id, idempotency_key: 'opti24_' || opti_tx.id,
              from_wallet_id: opti24_cards (баланс карт),
              to_wallet_id: ext_suppliers,
              lifecycle_status: 'approved',  // авто-апрув!
              settlement_status: 'completed',
              transaction_type: 'fuel_auto',
              actual_date: opti_tx.timestamp,
              description: 'Заправка ' || opti_tx.station_name || ', ' || opti_tx.liters || ' л'
            })
       d. UPDATE настройки: last_sync_at = now()
       e. Возврат отчёт: { count, total_amount, errors }
```

---

### Шаг 2. Обработка ошибок API

| Ошибка                                  | Действие                                                                          |
| --------------------------------------- | --------------------------------------------------------------------------------- |
| Опти24 недоступен                       | Не падаем, в audit_log отметка, retry через 5 минут                               |
| Карта не привязана к машине             | Транзакция в `fuel_transactions_raw` без `asset_id`, алерт админу «Привяжи карту» |
| Дубль (тот же external_id)              | Skip                                                                              |
| Невалидные данные (отрицательная сумма) | Лог в audit, skip, алерт                                                          |

## Точки отказа

| Что может пойти не так         | Что система делает                                         |
| ------------------------------ | ---------------------------------------------------------- |
| Опти24 API не отвечает         | Retry с backoff, алерт после 3 неудач                      |
| Изменился формат ответа Опти24 | Лог сырого ответа в `fuel_transactions_raw`, алерт         |
| Расход больше остатка по карте | Транзакция всё равно создаётся (карта может быть в минусе) |
| Двойной запуск синхронизации   | По idempotency_key — второй no-op                          |

---

## Что важно знать для реализации

1. **`fuel_transactions_raw`** — отдельная таблица для сырых данных от API. Это страховка: если что-то пошло не так, можно перепроанализировать.
2. **Внешние транзакции — auto-approved.** Мы не апрувим заправки от Опти24, потому что это факт от внешнего доверенного источника.
3. **Асимметрия с водительским ГСМ:** если водитель купил ГСМ за наличные — это draft (нужен апрув). Если по карте Опти24 — это auto-approved (не нужен апрув).
4. **Маппинг card → asset.** Хранится в таблице `fuel_cards`. Если водитель сменил машину — админ должен переключить привязку, иначе расход уйдёт на старую машину.
