## Суть

SaldaCargo — система управления транспортным бизнесом ИП Нигамедьянов А.С. (г. Верхняя Салда).
Автопарк ~12 единиц, направления: грузоперевозки + СТО. Цель: полный цифровой учёт.

## Компоненты системы

```
┌─────────────────────────────────────────────────────┐
│               MiniApp (MAX мессенджер)               │
│  Водитель → рейсы, заказы, расходы                  │
│  Механик → наряды, таймеры                          │
│  Администратор → ревью, финансы, дашборд            │
└──────────────────────┬──────────────────────────────┘
                       │ HTTPS API
┌──────────────────────▼──────────────────────────────┐
│             Next.js API Routes (Vercel)              │
│  createAdminClient() → bypasses RLS                  │
│  Cookie auth: salda_user_id                          │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────┐
│              Supabase (PostgreSQL)                   │
│  RLS включён, но API использует service_role key    │
│  ~25 таблиц, ENUMs, триггеры, pg_trgm               │
└─────────────────────────────────────────────────────┘
```

**WebApp (Dashboard):** планируется — аналитика и отчёты для владельца. Сейчас в разработке.

## Ключевые архитектурные решения

### 1. Двухосный статус транзакций

Каждая финансовая запись имеет ДВА независимых статуса:

| Ось            | Поле                | Значения                     | Смысл                         |
| -------------- | ------------------- | ---------------------------- | ----------------------------- |
| Жизненный цикл | `lifecycle_status`  | draft → approved → cancelled | Кто и когда подтвердил данные |
| Расчёт         | `settlement_status` | pending → completed          | Прошли ли деньги физически    |

**Балансы и P&L считаются ТОЛЬКО по `lifecycle_status='approved' AND settlement_status='completed'`.**

Сотрудник создаёт `draft`. Администратор переводит в `approved`. Внешние API (банк, Опти24) создают сразу `approved`.

### 2. Кастомная авторизация (не Supabase Auth)

Cookie `salda_user_id` = UUID из таблицы `users`.
Не используется Supabase JWT и `auth.uid()`.
Все API-роуты используют `createAdminClient()` (service_role) — обходят RLS.

Подробнее: [[Аутентификация]]

### 3. ЗП — только ручной ввод

ЗП водителя (`trip_orders.driver_pay`) вводится вручную при каждом заказе.
Система показывает подсказку (~30%), но **не пересчитывает автоматически**.
Автоматический триггер ЗП ЗАПРЕЩЁН — нарушит весь учёт.

### 4. Soft-delete везде

Никакого `DELETE` для бизнес-данных:

- Заказы: `lifecycle_status = 'cancelled'`
- Транзакции: `lifecycle_status = 'cancelled'` + `cancelled_reason`
- Сотрудники: `is_active = false`
- Исключение: `trip_expenses` — физическое удаление технически допустимо

### 5. Деньги только строкой

```typescript
// БД: DECIMAL(12,2) → "5000.00"
// TS: всегда string, никогда number
// UI: <Money amount="5000.00" /> → "5 000 ₽"
```

### 6. Idempotency Key

Каждая форма создания генерирует UUID при монтировании:

```typescript
const [key] = useState(() => crypto.randomUUID());
```

Сервер при дублировании (PostgreSQL 23505) возвращает существующую запись с 200.

## Принципы

1. **Сотрудник как поставщик данных** — водитель и механик создают только `draft`
2. **Единство истины** — балансы только из транзакций, никогда не хранятся статически
3. **Бережливость** — Supabase Free + Vercel Hobby до масштабирования
4. **Безопасность данных** — история не удаляется, только отменяется
