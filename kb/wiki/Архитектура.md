## Суть

SaldaCargo — система управления транспортным бизнесом ИП Нигамедьянов А.С. (г. Верхняя Салда).
Автопарк ~12 единиц, направления: грузоперевозки + СТО. Цель: полный цифровой учёт.

## Разделение ответственности: MiniApp vs WebApp

**Это ключевое архитектурное решение. Дублировать функции между приложениями — ЗАПРЕЩЕНО.**

### MiniApp — инструмент операций

MiniApp — это точка ввода данных. Всё, что делается быстро, в движении, с телефона:

- Создать рейс, добавить заказ, зафиксировать расход
- Завершить рейс с одометром
- Провести инкассацию (собрать наличные с водителя)
- Добавить доход или расход компании (произвольная транзакция)
- Одобрить / вернуть рейс на ревью
- Просмотреть оперативный дашборд (KPI за день)

Операции из MiniApp немедленно попадают в общую БД и автоматически отражаются в WebApp.

### WebApp Dashboard — инструмент анализа

WebApp — это витрина данных. Всё, что удобно смотреть на большом экране, в цифрах и графиках:

- P&L по месяцам, динамика выручки и расходов
- Журнал всех транзакций с фильтрами и поиском
- Дебиторка: кто, сколько, насколько просрочено
- Кредиты: тело, остаток, платёжный календарь
- Автопарк: состояние, одометры, ТО
- Персонал: подотчёт, ЗП, эффективность
- Рейсы на ревью — одобрить / вернуть (удобнее на ПК)
- **Постоянные клиенты** — аналитические карточки: выручка, рейсы, средний чек, долги, активность

**В WebApp НЕ нужно дублировать формы ввода, которые уже есть в MiniApp.**  
Если данные корректно записаны в БД через MiniApp — WebApp их увидит автоматически.

## Ключевые архитектурные решения

### 1. Двухосный статус транзакций

Каждая финансовая запись имеет ДВА независимых статуса:

| Ось            | Поле                | Значения                     | Смысл                         |
| -------------- | ------------------- | ---------------------------- | ----------------------------- |
| Жизненный цикл | `lifecycle_status`  | draft → approved → cancelled | Кто и когда подтвердил данные |
| Расчёт         | `settlement_status` | pending → completed          | Прошли ли деньги физически    |

**Балансы и P&L считаются ТОЛЬКО по `lifecycle_status='approved' AND settlement_status='completed'`.**

Сотрудник создаёт `draft`. Администратор переводит в `approved`. Внешние API (банк, Опти24) создают сразу `approved`.

### 2. Кастомная авторизация (не Supabase Auth)

Cookie `salda_user_id` = UUID из таблицы `users`.
Не используется Supabase JWT и `auth.uid()`.
Все API-роуты используют `createAdminClient()` (service_role) — обходят RLS.

Подробнее: [[Аутентификация]]

### 3. ЗП — только ручной ввод

ЗП водителя (`trip_orders.driver_pay`) вводится вручную при каждом заказе.
Система показывает подсказку (~30%), но **не пересчитывает автоматически**.
Автоматический триггер ЗП ЗАПРЕЩЁН — нарушит весь учёт.

### 4. Soft-delete везде

Никакого `DELETE` для бизнес-данных:

- Заказы: `lifecycle_status = 'cancelled'`
- Транзакции: `lifecycle_status = 'cancelled'` + `cancelled_reason`
- Сотрудники: `is_active = false`
- Исключение: `trip_expenses` — физическое удаление технически допустимо

### 5. Деньги только строкой

```typescript
// БД: DECIMAL(12,2) → "5000.00"
// TS: всегда string, никогда number
// UI: <Money amount="5000.00" /> → "5 000 ₽"
```

### 6. Idempotency Key

Каждая форма создания генерирует UUID при монтировании:

```typescript
const [key] = useState(() => crypto.randomUUID());
```

Сервер при дублировании (PostgreSQL 23505) возвращает существующую запись с 200.

## Domain-модули (`packages/domain/`)

Система спроектирована, частично реализована. Полное описание: [[docs/architecture/02-modules.md]].

### Почему не используется полностью

MiniApp строился быстро — логика писалась прямо в API-роутах. Это технический долг: система описана, код её не соблюдает. Не переписывать рабочий код, но и не накапливать долг дальше.

### Текущее состояние

| Пакет                            | Статус          | Что есть                                                  |
| -------------------------------- | --------------- | --------------------------------------------------------- |
| `@saldacargo/shared`             | ✅ Используется | formatDate, formatMoney, addMoney, generateIdempotencyKey |
| `@saldacargo/domain-finance`     | ✅ Используется | calculateWalletBalance, getPaymentMethodLabel             |
| `@saldacargo/domain-service`     | ✅ Частично     | getMechanicSummary, startWork, stopWork                   |
| `@saldacargo/domain-logistics`   | ❌ Пустой       | —                                                         |
| `@saldacargo/domain-fleet`       | ❌ Пустой       | —                                                         |
| `@saldacargo/domain-payroll`     | ❌ Пустой       | —                                                         |
| `@saldacargo/domain-receivables` | ❌ Пустой       | —                                                         |

### Правило: когда писать в domain, когда в роут

**В роут** — простой CRUD, используемый в одном месте:

```typescript
// Простой SELECT → оставить в роуте
supabase.from('loans').select('*').order('created_at');
```

**В domain** — при любом из трёх условий:

1. Та же логика нужна и в miniapp, и в webapp
2. Это вычисление (не SELECT): расчёт баланса, просрочки, P&L
3. Это бизнес-правило которое хочется протестировать без HTTP

```typescript
// packages/domain/receivables/src/index.ts
export function calculateOverdueDays(dueDate: string): number { ... }

// packages/domain/finance/src/index.ts
export function calculateDriverBalance(transactions, driverId): string { ... }
```

**Направление импортов строго сверху вниз** (apps → domain → shared). Подробнее: [[docs/architecture/02-modules.md]].

## Принципы

1. **Сотрудник как поставщик данных** — водитель и механик создают только `draft`
2. **Единство истины** — балансы только из транзакций, никогда не хранятся статически
3. **Бережливость** — Supabase Free + Vercel Hobby до масштабирования
4. **Безопасность данных** — история не удаляется, только отменяется
