# Миграции БД

**Теги:** #бд #sql #devops #миграции

В SaldaCargo используется механизм миграций для внесения любых изменений в схему базы данных. Это гарантирует консистентность между локальной средой разработки и production-сервером.

## Процесс создания миграции

1. **Файл:** Миграция создается в папке `supabase/migrations/`.
2. **Имя:** `<timestamp>_<name>.sql` (например, `20260430120000_add_mechanic_logic.sql`).
3. **Содержимое:** Чистый SQL код (DDL и DML).
4. **Типизация:** После применения миграции **обязательно** нужно обновить типы в проекте.
   Команда `pnpm db:types` работает только с локальным Docker (который обычно не запущен).
   Правильная команда для генерации из **удалённого** Supabase:
   ```bash
   npx supabase gen types typescript --project-id dzwuvbhfqnsxbfbynmgz > packages/shared-types/src/database.types.ts
   ```
   Project ID `dzwuvbhfqnsxbfbynmgz` — из `apps/web/.env.local` (переменная `SUPABASE_PROJECT_REF`).

## Правила написания миграций

- **Идемпотентность:** Используйте `CREATE TABLE IF NOT EXISTS` или `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`, где это уместно.
- **RLS:** Каждая новая таблица должна сопровождаться политиками Row Level Security (RLS).
- **Комментарии:** Добавляйте `COMMENT ON TABLE` и `COMMENT ON COLUMN` для документирования полей прямо в БД.
- **Откат:** В теле миграции (или в отдельном файле) всегда описывайте SQL для отката изменений.

## Пример сложной миграции (СТО)

Расширение логики механика включало в себя:

- Создание каталога работ (`work_catalog`).
- Внедрение системы таймеров через `work_time_logs`.
- Создание триггеров для автоматического расчета фактического времени работы (`actual_minutes`).
- Настройку политик RLS, позволяющих механику видеть только свои активные наряды.
