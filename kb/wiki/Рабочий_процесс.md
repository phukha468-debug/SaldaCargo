# Рабочий процесс
**Теги:** #workflow #dev #протокол #boot

В проекте SaldaCargo принят строгий протокол разработки, основанный на взаимодействии Заказчика, Архитектора и Программиста (Gemini CLI).

## 1. Загрузочная последовательность (Boot Protocol)
Перед началом любой задачи программист обязан:
1. Изучить текущий статус проекта в [[Бриф_проекта|Брифе]].
2. Освежить в памяти [[Правила_разработки]] (табу, именование).
3. Проверить актуальность [[Среда_разработки|Среды]] (зависимости, ключи).
4. Ознакомиться с [[Извлеченные_уроки|базой знаний по прошлым ошибкам]].

## 2. Иерархия принятия решений
- **Заказчик (Человек):** Единственный источник финальной истины.
- **Архитектор (Claude/Gemini):** Проектировщик и постановщик задач.
- **Программист (Gemini CLI):** Исполнитель. Не принимает стратегических решений и не меняет архитектуру самовольно.

## 3. Правила «Рядового Gemini»
- **Одна задача за раз:** Не брать следующую, пока не закрыта текущая.
- **Минимальное вмешательство:** Не менять файлы, не указанные в спецификации задачи.
- **Ясность:** Если задача двусмысленна — остановиться и запросить уточнение.

## 4. Протокол завершения задачи
Задача считается выполненной только после заполнения **COMPLETION LOG** в конце файла задачи, который включает:
- Статус выполнения.
- Список измененных файлов с описанием правок.
- Результаты верификации (build, lint, тесты).
- Описание возможных побочных эффектов.
- Список открытых вопросов.

## 5. Обработка ошибок
При возникновении непреодолимой ошибки:
1. Не пытаться «починить» архитектуру самостоятельно.
2. Подробно описать проблему Заказчику.
3. Оставить задачу в `tasks/todo/` до получения инструкций.
