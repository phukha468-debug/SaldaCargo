Описание системы авторизации в обоих приложениях. **Это не Supabase Auth** — кастомная схема на куках.

## Как работает вход

### Схема: выбор из списка (PIN-free MVP)

Экран `/` (`app/page.tsx`) — `RootDispatcher` — основной диспетчер сессии.

**Шаги при первом входе:**

1. Пользователь выбирает роль (водитель / механик / администратор)
2. Загружается список сотрудников `/api/users/public?role={role}`
3. Пользователь выбирает себя из списка
4. Клиент устанавливает cookie: `document.cookie = 'salda_user_id={users.id}; path=/; max-age=604800'`
5. В `localStorage` сохраняется `selected_role` и (для водителей) `active_vehicle_id`
6. Редирект на `/driver`, `/mechanic` или `/admin`

**Восстановление сессии (после смахивания/перезапуска):**

```typescript
// При монтировании RootDispatcher:
const existingUserId = getCookieValue('salda_user_id'); // читаем из document.cookie
const savedRole = localStorage.getItem('selected_role');
const savedVehicleId = localStorage.getItem('active_vehicle_id');

if (existingUserId && savedRole) {
  if (savedRole === 'driver' && savedVehicleId) router.replace('/driver');
  else if (savedRole === 'admin' || savedRole === 'owner') router.replace('/admin');
  else if (savedRole === 'mechanic') router.replace('/mechanic');
}
```

Cookie TTL: 7 дней (`max-age=604800`). Сессия восстанавливается автоматически.

## Middleware (`apps/miniapp/middleware.ts`)

```typescript
// Защита всех маршрутов кроме / и /api/auth/* и публичных API
if (!userId && !isAuthPage && !isAuthApi && !isPublicApi && pathname !== '/') {
  return NextResponse.redirect(new URL('/login', request.url));
}
```

Публичные маршруты (без авторизации):

- `/` — RootDispatcher
- `/api/auth/*` — авторизация
- `/api/users/public` — список пользователей для выбора
- `/api/vehicles/public` — список машин для выбора

Защищённые:

- `/driver/*`, `/mechanic/*`, `/admin/*`
- Все `/api/*` кроме публичных

## API авторизации

### `POST /api/auth/max`

Авторизация через MAX Mini App. Принимает `initData.max_user_id`, ищет в `users.max_user_id`.

```json
Body:  { "initData": { "max_user_id": "..." } }
OK:    { "user": { "id", "name", "roles" } }
403:   { "error": "Доступ запрещён. Ваш MAX ID: ..." }
```

### `GET /api/users/public?role={role}`

Возвращает список активных пользователей по роли. Используется для экрана выбора себя.

### `GET /api/vehicles/public`

Возвращает список активных машин (`status = 'active'`). Для экрана выбора машины водителем.

### `POST /api/auth/logout`

Очищает cookie. Используется при выходе.

## Как читать userId в API-роутах

```typescript
import { cookies } from 'next/headers';

const cookieStore = await cookies();
const userId = cookieStore.get('salda_user_id')?.value;
if (!userId) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
```

Используется в: `/api/trips/route.ts`, `/api/driver/finance`, `/api/driver/profile`, `/api/admin/transactions`, `/api/auth/logout`.

## Supabase Admin Client

Все API-роуты используют `createAdminClient()` (service_role key), который **обходит RLS**.
Это сделано намеренно: пользователи в MiniApp не имеют Supabase-сессии (нет JWT).

```typescript
// lib/supabase/admin.ts
import { createClient } from '@supabase/supabase-js';

export function createAdminClient() {
  return createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.SUPABASE_SERVICE_ROLE_KEY!, // server-only
  );
}
```

> ⚠️ `SUPABASE_SERVICE_ROLE_KEY` — только на сервере (в API routes). Никогда не на клиенте.

## Переменные окружения

| Переменная                      | Где используется                 |
| ------------------------------- | -------------------------------- |
| `NEXT_PUBLIC_SUPABASE_URL`      | Client + Server                  |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Client (не используется активно) |
| `SUPABASE_SERVICE_ROLE_KEY`     | Server only (Admin Client)       |

## Известные ограничения MVP

1. **Нет пароля/PIN** — любой, кто знает имя из списка, может войти
2. **Нет HMAC-верификации MAX** — проверка подписи Mini App не реализована
3. **Нет ролевой защиты в middleware** — водитель технически может зайти на `/admin`
4. **Cookie без `httpOnly`** — уязвим к XSS (приемлемо для внутреннего приложения)

## WebApp авторизация (apps/web)

Отдельная схема от MiniApp. Cookie: `salda_auth_token` = `users.id`.

### Вход

`POST /api/auth/login` — принимает `{ phone, pin }`, проверяет `users.phone` + `users.pin_code` из БД.  
Доступ только для ролей `admin` и `owner`. Также проверяется `users.is_active = true`.

Cookie устанавливается с `httpOnly: true`, `sameSite: 'lax'`, `secure: true` (в production), `maxAge: 7 дней`.

Страница `/login` — публичная. После входа редирект на `?from=` (страница, куда пытались зайти).

### Middleware (`apps/web/middleware.ts`)

Защищает все маршруты кроме `/login`, `/auth/error`, `/auth/callback`, `/api/auth/*`, `/api/cron/*`.

- Страницы без токена → редирект на `/login?from={pathname}`
- API-роуты без токена → `401 { error: 'Unauthorized' }`

### Как читать userId в WebApp API-роутах

```typescript
import { requireAuth } from '@/lib/auth';

const auth = await requireAuth();
if (auth instanceof NextResponse) return auth;
// auth.userId доступен
```

Либо напрямую:

```typescript
import { cookies } from 'next/headers';
const cookieStore = await cookies();
const userId = cookieStore.get('salda_auth_token')?.value;
if (!userId) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
```

> ⚠️ **Не путать куки:** MiniApp использует `salda_user_id`, WebApp — `salda_auth_token`.
