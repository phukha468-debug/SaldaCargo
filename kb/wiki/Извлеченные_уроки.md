# Извлеченные уроки
**Теги:** #опыт #база_знаний #lessons #ошибки

Этот документ является живым реестром ошибок, совершенных в ходе разработки, и правил, которые помогают избежать их в будущем.

## Процесс и методология (LL-PROC)
- **LL-PROC-001: Completion Log обязателен.** Каждая задача должна завершаться подробным отчетом. Без него невозможно восстановить контекст при возвращении к коду через время.
- **LL-PROC-002: Конкретика в верификации.** Запрещено использовать размытые критерии типа «проверь, что работает». Каждый таск должен содержать четкие шаги проверки (входные данные -> ожидаемый результат).
- **LL-PROC-003: Правило трех фиксов.** Если подсистема требует третьего патча подряд — это повод остановиться и провести архитектурное ревью (overhaul), вместо того чтобы продолжать лечить симптомы.

## Разработка и Данные
- **Асинхронные операции:** Запрещено выполнять навигацию (`router.push`) до подтверждения записи данных в БД.
- **Обработка ошибок:** Каждый `async`-вызов обязан быть обернут в `try/catch` с логированием контекста.
- **Финансы:** Никогда не хранить балансы. Только расчет из транзакций. Использование `float` для денег — критическая ошибка.

## Работа с внешними системами
- **Интеграции:** Все внешние данные (Опти24, Wialon) должны сначала попадать в «сырые» таблицы (`_raw`) перед обработкой и матчингом с основной системой.
