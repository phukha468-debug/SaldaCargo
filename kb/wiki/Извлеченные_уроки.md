# Извлеченные уроки

**Теги:** #опыт #база_знаний #lessons #ошибки
**Обновлено:** 2026-06-01

Живой реестр ошибок и правил, выведенных в ходе разработки. Читать перед любой задачей.

## Процесс и методология (LL-PROC)

- **LL-PROC-001: Completion Log обязателен.** Каждая задача должна завершаться подробным отчётом. Без него невозможно восстановить контекст при возвращении к коду.
- **LL-PROC-002: Конкретика в верификации.** Запрещено использовать размытые критерии типа «проверь, что работает». Чёткие шаги: входные данные → ожидаемый результат.
- **LL-PROC-003: Правило трёх фиксов.** Если подсистема требует третьего патча подряд — это повод остановиться и провести архитектурное ревью, а не продолжать лечить симптомы.

## TypeScript и Supabase

- **LL-TS-001: `as any` только на `.from()`.**
  Правильно: `(supabase.from('trips') as any).update(body)`
  Неправильно: `supabase.from('trips').update(body) as any`
  Если `as any` в конце цепочки — TypeScript всё равно проверяет аргументы `.update()` / `.insert()`.
  Затрагивало: orders/route.ts, profile/route.ts, transactions/route.ts.

- **LL-TS-002: Результат `RegExp.match()` может быть undefined.**
  Правильно: `match?.[2]` вместо `match[2]`.
  В `app/page.tsx` при парсинге cookie через regex.

- **LL-TS-003: `catch { }` вместо `catch (_e) { }`.**
  Линтер ругается на неиспользуемую переменную `_e`. Используй пустой `catch {}` если ошибка не нужна.

- **LL-TS-004: ESLint-комментарий только для реальных правил.**
  `// eslint-disable-next-line react-hooks/exhaustive-deps` вызывает ошибку если правило не настроено в конфиге. Проверяй `.eslintrc` перед добавлением таких комментариев.

## Supabase и PostgreSQL

- **LL-DB-001: GRANT ≠ RLS.**
  У таблицы `trip_expenses` не было GRANT на DELETE для `service_role`. RLS был настроен, но PostgreSQL GRANT не выдан.
  Симптом: `permission denied for table trip_expenses` даже через Admin Client.
  Решение: `GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO anon, authenticated, service_role;`

- **LL-DB-002: Admin Client обходит RLS, но не GRANT.**
  `createAdminClient()` использует service_role key и обходит политики RLS.
  Однако если таблица создана без явного GRANT, даже service_role может получить отказ.

- **LL-DB-003: `maybeSingle()` vs `single()`.**
  `single()` бросает ошибку если записей 0 или > 1.
  `maybeSingle()` возвращает `null` если запись не найдена (не ошибку).
  Используй `maybeSingle()` когда запись может отсутствовать.

- **LL-DB-004: Триггер одометра.**
  `trg_trips_sync_odometer` срабатывает только при `UPDATE trips.odometer_end`.
  Если `odometer_end` устанавливается при INSERT — триггер не срабатывает.
  Решение: всегда обновлять через отдельный PATCH/UPDATE.

## CSS и компоновка

- **LL-CSS-001: CSS Grid и коллапсируемые тайлы — `items-start` обязателен.**
  Дефолтное поведение Grid — `align-items: stretch` — все ячейки в строке растягиваются до высоты самой высокой.
  Если один тайл раскрыт, все соседи в той же строке становятся такой же высоты и визуально «открываются».
  Решение: добавить `items-start` на контейнер grid (`grid-cols-3 gap-2 items-start`).

- **LL-CSS-002: Выравнивание колонок метрик между карточками.**
  Чтобы «Выручка» одной карточки была точно под «Выручкой» другой, нужно зафиксировать ширину метрик:
  - Правильно: `shrink-0` на блок метрик + фиксированная ширина `w-[100px]` на каждую колонку
  - Неправильно: `flex-1 justify-center` — блок метрик сдвигается вместе с левым контентом
  - Левая часть при этом должна быть `flex-1 min-w-0`

## Архитектура и паттерны

- **LL-ARCH-001: Подотчёт из trip_orders, не из wallets.**
  Изначально подотчёт водителя брался из таблицы `wallets/transactions` — она требует одобрения администратора.
  Правильно: считать напрямую из `trip_orders WHERE payment_method = 'cash' AND lifecycle_status != 'cancelled'`.
  Водитель получает нал сразу — не нужно ждать ревью.

- **LL-ARCH-007: Подотчёт только `cash`, не `card_driver`.**
  `card_driver` = корпоративная карта. Деньги поступают не на руки водителю — он не несёт за них ответственность.
  В подотчёт входит только `payment_method = 'cash'`.
  Затронуто в сессии 11: `driver/finance`, `admin/cash-collections`, `staff/settle`.

- **LL-ARCH-008: Единый фильтр выручки — обязательно `trips.lifecycle_status = 'approved'`.**
  Finance и Counterparties должны считать выручку по одним и тем же правилам:
  `trip_orders.lifecycle=approved AND settlement=completed AND trips.lifecycle=approved`.
  Без последнего условия возвращённые рейсы могут попасть в статистику.

- **LL-ARCH-002: MOCK_DRIVER_ID — антипаттерн.**
  Хардкод ID водителя в API-роутах → все видят одни данные.
  Правильно: читать `salda_user_id` из cookie через `cookies()` из `next/headers`.

- **LL-ARCH-003: Сессия должна восстанавливаться.**
  Начальное состояние `step = 'role'` → каждый раз экран выбора.
  Правильно: `step = 'restoring'`, в `useEffect` проверяем cookie + localStorage, делаем redirect.

- **LL-ARCH-004: Invalidate перед navigate.**
  После добавления заказа/расхода нужно `invalidateQueries` ДО `router.push()`.
  Иначе страница рейса покажет старые данные до обновления кэша.

- **LL-ARCH-006: Двухшаговое подтверждение удаления (паттерн).**
  Для деструктивных действий в UI: первый клик меняет кнопку на «Ещё раз — удалить!» + `animate-pulse`,
  второй подтверждает. Таймер `setTimeout` на 4 сек автоматически сбрасывает состояние.
  `useRef` для хранения таймера чтобы очищать при размонтировании.
  Используется в `/review` и `/fleet`.

- **LL-ARCH-005: `staleTime: 0` для страниц ревью.**
  Страница деталей рейса для ревью должна всегда показывать свежие данные.
  Используй `staleTime: 0` чтобы TanStack Query всегда делал fetch при монтировании.

- **LL-ARCH-010: `refetchQueries` vs `invalidateQueries` в TanStack Query.**
  `queryClient.refetchQueries()` — немедленный blocking рефетч, перерисовывает компонент прямо сейчас.
  `queryClient.invalidateQueries()` — помечает кеш устаревшим, рефетч происходит фоново.
  На медленных операциях (добавление работы в наряд) `refetchQueries` блокировало UI на ~500мс.
  **Правило:** для мутаций использовать `invalidateQueries`. `refetchQueries` — только если нужны данные немедленно до следующего рендера.
  Затронуто: `apps/web/app/(dashboard)/garage/page.tsx` сессия 16.

- **LL-ARCH-011: `staleTime` предотвращает лишние рефетчи при фокусе окна.**
  По умолчанию TanStack Query делает рефетч при каждом `window focus`. Для тяжёлых запросов (список 95 работ) это заметно.
  Решение: `staleTime: 30000` (30 сек) — данные считаются свежими, рефетч не нужен.
  Не ставить `staleTime` большим для данных с часто меняющимся состоянием (статусы нарядов/рейсов).

- **LL-ARCH-012: Умная инвалидация — инвалидировать только то что изменилось.**
  Не нужно инвалидировать `garage-orders` и `garage-dashboard` при каждом PATCH на наряд.
  Эти страницы меняются только когда меняется `lifecycle_status`.
  Паттерн: `if ('lifecycle_status' in variables) { invalidate dashboard }` — проверять переменные мутации.

- **LL-ARCH-013: Зачёт аванса без новой таблицы — паттерн ADVANCE_OFFSET.**
  При зачёте аванса в счёт ЗП не нужна отдельная таблица или поле.
  Используй `ADVANCE_PAYMENT income` транзакцию: `advance_balance = SUM(expense) - SUM(income)` по категории ADVANCE_PAYMENT.
  Всегда on-the-fly, не хранить баланс статически. UUID категории: `a0000000-0000-0000-0000-000000000001`.

- **LL-ARCH-014: FK constraint порядок удаления в PostgreSQL.**
  При очистке данных через `DELETE` нужно соблюдать порядок:
  1. `DELETE FROM work_time_logs` (ссылается на service_order_works)
  2. `DELETE FROM service_order_works`
  3. `DELETE FROM service_orders`
     Иначе: `ERROR 23503: update or delete on table "service_order_works" violates foreign key constraint`.

- **LL-ARCH-015: Проверка состояния React-формы vs сохранённых данных в БД.**
  Симптом: механик выбран в dropdown, но кнопка «Назначьте механика» всё равно заблокирована.
  Причина: `order.mechanic` — сохранённые данные из БД. `editMechanic` — локальный state dropdown.
  До нажатия «Сохранить» `order.mechanic` не обновляется. Условие должно проверять оба источника:
  `!order.mechanic && !(editMechanic && editMechanic !== '')`

- **LL-ARCH-009: Единая формула подотчёта водителя в обоих приложениях.**
  Подотчёт = `SUM(cash-заказы, не cancelled) − SUM(cash-расходы) − SUM(инкассации)`.
  Формула не зависит от lifecycle_status (нал у водителя физически есть, неважно одобрил админ рейс или нет).
  При выплате ЗП водителю — авто-закрытие подотчёта (cash_collection на остаток) в обоих staff-settle роутах.

## Мобильный UX

- **LL-UX-001: Клавиатура перекрывает поля ввода.**
  Фиксированные кнопки внизу (`fixed bottom-0`) + числовая клавиатура = поле не видно.
  Решения:
  1. Убрать `fixed`, дать кнопке быть частью скроллируемой формы
  2. `onFocus: setTimeout(() => e.target.scrollIntoView({ behavior: 'smooth', block: 'center' }), 300)`

- **LL-UX-002: Защита от двойного нажатия обязательна.**
  При медленном интернете пользователь может нажать "Сохранить" несколько раз.
  Паттерн:
  ```typescript
  const [submitted, setSubmitted] = useState(false);
  const handleSubmit = async () => {
    if (submitting || submitted) return;
    setSubmitted(true);
    // ...
  };
  ```
  Дополнительно: `idempotency_key` на сервере.

## Разработка и данные

- **LL-DEV-004: Пустая строка UUID → NULL в Supabase.**
  При отправке `category_id: ''` (пустая строка) в Supabase для колонки типа UUID — значение конвертируется в NULL.
  Если колонка NOT NULL — PostgreSQL выбросит ошибку `null value in column "X" violates not-null constraint`.
  Решение: всегда добавлять серверную валидацию `if (!body.category_id) return 400` ДО вставки.
  Симптом встречался в: `/api/admin/transactions`, `/api/wallets/set-balance`.

- **LL-DEV-005: INSERT в `transactions` — `category_id` обязателен явно.**
  Колонка `category_id` в таблице `transactions` — NOT NULL. При любом INSERT передавать явно.
  Особенно внимательно: корректировочные транзакции (set-balance), переводы, системные записи —
  легко забыть добавить category_id когда пишешь "служебный" INSERT.

- **LL-DEV-001: Асинхронные операции.** Навигация (`router.push`) только после подтверждения записи в БД.
- **LL-DEV-002: Финансы.** Никогда `number` для денег. Только `string` с DECIMAL(12,2) в БД.
- **LL-DEV-003: Мягкое удаление.** `DELETE` запрещён для бизнес-данных. Только `lifecycle_status='cancelled'` с причиной. Исключение: `trip_expenses` (технически допустимо, т.к. нет lifecycle).

## AI-генерация данных (LL-AI)

- **LL-AI-001: Нормализация `machine_type` из AI-ответа.**
  ИИ (Claude, ChatGPT) иногда генерирует `"company"` вместо `"own"` для поля `machine_type`.
  Enum в БД принимает ТОЛЬКО `'own'` и `'client'`. При парсинге AI-JSON обязательно нормализовывать:

  ```typescript
  if (!['own', 'client'].includes(parsed.order.machine_type)) {
    parsed.order.machine_type = 'own';
  }
  ```

  Местонахождение: `tryParseJson` в `apps/web/app/(dashboard)/garage/page.tsx`.

- **LL-AI-002: Нормализация `priority` из AI-ответа.**
  ИИ может генерировать `"high"`, `"critical"`, `"medium"` и т.п.
  CHECK constraint в БД: `priority IN ('low', 'normal', 'urgent')`.
  Нормализация в `tryParseJson` + server-side fallback в import route:

  ```typescript
  priority: ['low', 'normal', 'urgent'].includes(body.order.priority ?? '')
    ? body.order.priority
    : 'normal';
  ```

- **LL-AI-003: Явные ограничения в system prompt важнее описания.**
  Вместо «machine_type: "own" — машина из парка компании» писать:
  «ТОЛЬКО два значения — "client" или "own". Никаких других ("company", "fleet" и т.п.)».
  Запрет конкретных вариантов эффективнее позитивного описания.

## React UI-паттерны (LL-UI)

- **LL-UI-001: Backdrop drag-close баг.**
  Сценарий: пользователь нажимает внутри модалки, тащит мышь на backdrop, отпускает — браузер стреляет `click` на backdrop, модалка закрывается.
  `stopPropagation` на inner panel не помогает (click генерируется на месте mouseup).
  **Правильный фикс:**

  ```typescript
  const backdropMouseDownOnSelf = useRef(false);
  // На backdrop:
  onMouseDown={(e) => { backdropMouseDownOnSelf.current = e.target === e.currentTarget; }}
  onClick={(e) => { if (backdropMouseDownOnSelf.current && e.target === e.currentTarget) onClose(); }}
  ```

  Использовать для всех модалок где backdrop кликабелен.

- **LL-UI-002: Мгновенные клиентские фильтры vs серверные запросы.**
  Для списков где данные уже загружены — фильтровать client-side через `useMemo`. Нет задержек, нет лишних запросов.
  Хорошо подходит для: заказ-нарядов, транзакций, сотрудников (< 1000 записей).
  Паттерн: один широкий запрос (`?filter=all`) + `useMemo` для производных списков.
  Обновление раз в 60 сек (`staleTime: 60000, refetchInterval: 60000`) вместо при каждом взаимодействии.

- **LL-UI-003: Числовые поля — очистка нуля при фокусе.**
  Если поле инициализировано `'0'` (расчётное значение), пользователь ожидает начать вводить с чистого поля.
  ```typescript
  onFocus={(e) => { if (e.target.value === '0') setClosePay1(''); }}
  ```
  Применять для полей ЗП, сумм, коэффициентов.

## Финансовая логика (LL-FIN)

- **LL-FIN-001: Payroll-транзакции наряда ОБЯЗАНЫ содержать `service_order_id`.**
  Webapp-апруве нарядов (`/api/garage/orders/[id]`) создавал payroll-транзакции
  без `service_order_id`. Из-за этого при "открытии заново" наряда
  (`WHERE service_order_id = id`) старые транзакции не удалялись — ЗП двоилась.
  Правило: любая транзакция, связанная с нарядом, ДОЛЖНА иметь `service_order_id`.

- **LL-FIN-002: При возврате наряда (`returned`) — чистить финансы.**
  Порядок действий при `lifecycle_status = 'returned'`:
  1. `DELETE FROM transactions WHERE service_order_id = id`
  2. `UPDATE service_order_works SET salary_paid = false WHERE service_order_id = id`
  3. Только потом обновить lifecycle наряда
     Без шагов 1-2 повторный апрув создаст дублирующие транзакции.

- **LL-FIN-003: QR-оплата → Расчётный счёт, Наличные/Карта → Касса.**
  При апруве рейса авто-транзакции:
  - `payment_method = 'cash'` или `'card_driver'` → `to_wallet_id = CASH_ID`
  - `payment_method = 'qr'` → `to_wallet_id = BANK_ID`
  - `payment_method = 'debt_cash'` → транзакция НЕ создаётся (идёт в дебиторку)

- **LL-FIN-004: Долг юрлица → Р/С, долг физлица → Касса.**
  При погашении дебиторки маршрутизация зависит от `counterparties.is_legal_entity`:
  - `true` (юрлицо) → Расчётный счёт
  - `false` (физлицо) → Касса
    Поле `is_legal_entity` добавлено миграцией `20260601000001`.
