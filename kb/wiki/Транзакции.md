# Транзакции

**Теги:** #финансы #бд #концепция
**Обновлено:** 2026-05-20

## Суть

Транзакции — единственный источник истины для финансового состояния компании. Любое изменение баланса (выручка, расход, ЗП, инкассация, погашение долга) является транзакцией в таблице `transactions`.

---

## Двухосный статус

Каждая транзакция и каждый заказ (`trip_orders`) имеют два независимых статуса:

### Ось 1 — Lifecycle (кто утвердил)

| Статус      | Значение                                                |
| ----------- | ------------------------------------------------------- |
| `draft`     | Создано сотрудником (водитель, механик) — ждёт проверки |
| `approved`  | Подтверждено администратором. Влияет на балансы и P&L   |
| `returned`  | Возвращено на доработку (только рейсы)                  |
| `cancelled` | Аннулировано (с причиной). Из расчётов исключается      |

### Ось 2 — Settlement (прошли ли деньги физически)

| Статус      | Значение                                            |
| ----------- | --------------------------------------------------- |
| `pending`   | Деньги ещё не поступили — дебиторская задолженность |
| `completed` | Деньги зачислены в кошелёк                          |

### Формула баланса

```sql
WHERE lifecycle_status = 'approved' AND settlement_status = 'completed'
```

**Балансы P&L и кошельки считаются ТОЛЬКО по этому условию.**  
`pending` — это дебиторка, она отдельно.

## Три кошелька компании

| UUID (последние 3 цифры) | Полный UUID                            | Название        | Тип             |
| ------------------------ | -------------------------------------- | --------------- | --------------- |
| `...001`                 | `10000000-0000-0000-0000-000000000001` | Расчётный счёт  | `bank_account`  |
| `...002`                 | `10000000-0000-0000-0000-000000000002` | Сейф (Наличные) | `cash_register` |
| `...003`                 | `10000000-0000-0000-0000-000000000003` | Карта           | `company_card`  |

## Способы оплаты заказа и их финансовый путь

Водитель при добавлении заказа выбирает один из 4 способов. Каждый способ определяет `settlement_status` заказа и конечный кошелёк при подтверждении.

| Способ        | Лейбл       | settlement при создании | Куда поступают деньги при подтверждении     |
| ------------- | ----------- | ----------------------- | ------------------------------------------- |
| `cash`        | 💵 Нал      | `completed`             | Подотчёт водителя → Сейф (через инкассацию) |
| `qr`          | 📱 QR / Р/С | `pending`               | Расчётный счёт (`...001`)                   |
| `card_driver` | 💳 На карту | `pending`               | Карта (`...003`)                            |
| `debt_cash`   | ⏳ Долг нал | `pending`               | Сейф (`...002`)                             |

### Детали по каждому способу

**`cash` — Наличные**

- Создаётся сразу `completed` — деньги есть, но у водителя "в кармане".
- Попадают в подотчёт водителя (`wallet_type = 'driver_accountable'`).
- Каждый вечер после смены водитель сдаёт наличные в офис — **инкассация**.
- Инкассация = `transfer` транзакция: подотчёт водителя → Сейф.

**`qr` — QR-оплата / Расчётный счёт**

- Создаётся `pending` — деньги ещё не пришли.
- Попадает в дебиторку.
- Есть два подвида (различаются клиентом):
  - **Физлицо** — оплата быстрая (день-два), QR-код.
  - **Юрлицо (договор)** — оплата после получения документов, срок 2–30 дней. _Отдельный учёт due_date ещё не реализован._
- При подтверждении: деньги → Расчётный счёт.

**`card_driver` — На карту (карта владельца)**

- Создаётся `pending`.
- Клиент переводит на личную дебетовую карту владельца.
- Попадает в дебиторку. Владелец подтверждает поступление.
- При подтверждении: деньги → Карта.

**`debt_cash` — Долг наличными**

- Создаётся `pending` — клиент не заплатил, обещал потом.
- Попадает в дебиторку.
- При подтверждении: деньги → Сейф (предполагается получение наличными).

## Жизненный цикл денег: от заказа до кошелька

```
Водитель добавляет заказ
    ↓
trip_orders.lifecycle_status = 'draft'
trip_orders.settlement_status = 'pending' (qr/card/debt) или 'completed' (cash)

Администратор одобряет рейс
    ↓
trips.lifecycle_status = 'approved'
trip_orders.lifecycle_status = 'approved'   ← деньги теперь "настоящие"

Для cash: сразу completed → видны в кошельке подотчёта водителя
          Инкассация переводит в Сейф

Для qr/card/debt_cash:
    settlement_status = 'pending' → попадают в дебиторку
    Администратор подтверждает оплату
         ↓
    trip_orders.settlement_status = 'completed'
    INSERT transactions (direction=income, to_wallet_id = нужный кошелёк)
         ↓
    Деньги появляются в балансе кошелька

--- Альтернативные ветки ---

Администратор возвращает рейс на доработку:
    trips.lifecycle_status = 'returned'
    trip_orders.lifecycle_status = 'draft'   ← заказы откатываются в черновик
    trips.ended_odometer = NULL              ← одометр сбрасывается
    Из расчётов и дебиторки исключается до повторного одобрения

Администратор аннулирует транзакцию:
    transactions.lifecycle_status = 'cancelled'
    transactions.cancelled_reason = '<причина>'
    Из всех расчётов исключается навсегда (soft-delete)
```

## Инкассация

Ежедневная операция передачи наличных от водителя в кассу.

- Тип транзакции: `transfer`
- `from_wallet_id` = кошелёк подотчёта водителя
- `to_wallet_id` = Сейф (`...002`)
- Категория: `CASH_COLLECT` (`72f677d4-...`)
- Создаётся администратором → сразу `approved + completed`

## Прямые транзакции администратора

Когда администратор вручную добавляет доход или расход через `/admin/finance`:

- `lifecycle_status = 'approved'` (сразу)
- `settlement_status = 'completed'` (сразу)
- Деньги немедленно видны в балансах

## Схема ЗП сотрудников: pending → completed

**Сессия 16 (20.05.2026).** Единая схема для всех категорий сотрудников: водители, грузчики, механики.

### Накопление долга

| Момент                             | Что создаётся                                                                |
| ---------------------------------- | ---------------------------------------------------------------------------- |
| Механик/Админ закрывает вид работы | `PAYROLL_MECHANIC`, `direction=expense`, `settlement_status=pending`         |
| Администратор апрувит рейс         | `PAYROLL_DRIVER` + `PAYROLL_LOADER` (если есть), `settlement_status=pending` |

`pending` — это **долг компании сотруднику**, деньги со счёта ещё не списаны.

### Выплата

Когда администратор нажимает «Выплатить» в разделе `/staff`:

1. Сервер читает все `pending` PAYROLL-транзакции сотрудника → `salary_total`
2. Считает `advance_balance` = сумма выданных авансов − сумма зачтённых
3. `offset = min(salary_total, advance_balance)` — зачёт аванса
4. `payout = salary_total − offset` — к выдаче деньгами
5. Все `pending` PAYROLL → `settlement_status = 'completed'`
6. Если `offset > 0` → INSERT `ADVANCE_PAYMENT` income (фиксирует погашение аванса)
7. Если `payout > 0` → нужен кошелёк; INSERT `expense` транзакция из выбранного кошелька

### API

- `GET /api/staff/payroll` — список сотрудников с `earned`, `paid`, `debt`, `advance_balance`, `payout`, `history`
- `GET /api/staff/settle?userId=X` — предварительный расчёт `{ salary_total, advance_balance, offset, payout }`
- `POST /api/staff/settle` — провести выплату

## ADVANCE_OFFSET паттерн (зачёт аванса в счёт ЗП)

**Проблема:** при выдаче аванса создаётся expense-транзакция `ADVANCE_PAYMENT`. При зачёте нужно зафиксировать факт погашения без новой таблицы.

**Решение:** `ADVANCE_PAYMENT income` — транзакция направления `income` с той же категорией `ADVANCE_PAYMENT`.

```
advance_balance = SUM(expense ADVANCE_PAYMENT) - SUM(income ADVANCE_PAYMENT)
```

Логика on-the-fly, никаких полей в таблице. Если `advance_balance > 0` — сотрудник должен компании.

## Типы операций

| direction  | Примеры                                                      |
| ---------- | ------------------------------------------------------------ |
| `income`   | Выручка с рейса, выручка СТО, погашение дебиторки            |
| `expense`  | ГСМ, запчасти, ЗП, аренда, налоги, страховки                 |
| `transfer` | Инкассация, перевод между кошельками, пополнение топл. карты |
